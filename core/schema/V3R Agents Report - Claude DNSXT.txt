## PART 1 — FRAMEWORK OVERVIEW

### 1.1 Why Multi-Agent

A single generalist AI produces generalist outputs. Generalist outputs on a constrained, specialized engineering system produce errors that cost 2+ hour rebuild cycles. The V3R multi-agent framework exists because each layer of the engineering process requires a different cognitive posture:

- Orchestration requires breadth and decomposition
- Research requires real-time sourcing and skepticism
- Code generation requires constraint awareness and pattern avoidance
- Debugging requires systematic elimination and zero assumption
- Documentation requires precision and forward-thinking
- Infrastructure requires cost awareness and security discipline

No single model excels at all six simultaneously. The framework assigns each posture to the model best suited for it and gates every output through a mandatory verification layer before it reaches the developer.

### 1.2 The Gate Rule

**Zero outputs bypass the System Debug Agent.** This is the single most important operational rule in the V3R framework. Every code block, every AWS command, every architectural recommendation passes through the debug gate before delivery. The gate exists because the cost of a bad output on this hardware and this infrastructure is not a quick fix — it is hours of lost time and potentially a broken production system.

### 1.3 Agent Interaction Model

```
Developer Request
       ↓
MODERATOR — decomposes, routes, validates
       ↓
┌──────────────────────────────────┐
│ KNOWLEDGE RETRIEVAL              │ ← sourced before code is written
│ CODER                            │ ← writes after KR confirms libraries
│ INFRASTRUCTURE                   │ ← pre-flight before any AWS change
└──────────────────────────────────┘
       ↓
SYSTEM DEBUG — mandatory gate
       ↓
DOCUMENTATION — inline comments added
       ↓
Developer Output
```

## PART 3 — CROSS-AGENT FAILURE ANALYSIS

### 3.1 The import os Failure Chain

This was the most significant multi-agent failure of the session and warrants a root cause analysis.

**What happened:**
1. Coder Agent wrote `GROQ_API_KEY = os.environ.get("GROQ_API_KEY", "")` without including `import os`
2. System Debug Agent cleared the output without catching the missing import
3. The code was delivered, deployed, and failed at runtime with NameError

**Root cause:** Two agents failed on the same item. The Coder Agent failed to apply basic Python hygiene (every stdlib module must be explicitly imported). The System Debug Agent failed to execute its import-usage consistency check.

**Resolution:** import os added, file rebuilt, deployed.

**Prevention:** System Debug Agent checklist must include explicit step: "For every name used in the file, confirm a corresponding import exists." This is now a mandatory gate item.

### 3.2 The PowerShell JSON Failure Chain

**What happened:**
1. Coder Agent produced inline JSON in PowerShell AWS CLI commands
2. System Debug Agent cleared the commands
3. Commands failed at execution with JSON parse errors
4. Pattern was corrected once, then repeated again
5. Multiple correction cycles before the pattern was formalized in the registry

**Root cause:** Neither Coder Agent nor System Debug Agent had the PowerShell JSON behavior in their constraint models. The failure mode was not anticipated before it occurred — it was learned through failure.

**Resolution:** WriteAllText method confirmed and added to Forbidden Patterns Registry with full rationale.

**Prevention:** Forbidden Patterns Registry is now the first check in System Debug gate. Any PowerShell command involving JSON must use WriteAllText to file before referencing file:// path.
