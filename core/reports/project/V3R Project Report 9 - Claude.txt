SECTION 1: PROJECT OVERVIEW AND VISION
================================================================================

V3R is a Cloud-Native GNN-integrated Multi-Agent Multi-LLM Industrial Science
and Intelligence Engineering Platform. It is not a chatbot. It is not a tool.
It is the foundational architecture for an AI system that accepts raw ideas as
input and outputs fully engineered products.

The core problem V3R solves is AI amnesia — the complete absence of persistent
memory in every current AI system. Every standard AI session starts from zero.
No history. No accumulated context. No engineering memory. V3R's persistent
memory layer eliminates this permanently. Knowledge ingested today is retrieved
tomorrow. Conversations accumulate into a growing intelligence base. Engineering
decisions are remembered across sessions. The system becomes more capable with
every interaction. This is not a feature. It is the entire architectural thesis.

V3R is being built entirely on free-tier AWS infrastructure by a single developer
operating on constrained personal hardware with no admin privileges. This is not
a limitation. It is proof of concept — demonstrating that industrial-scale AI
engineering can be architected, deployed, and operated without enterprise budget.

THE RAG SYSTEM WORKS BY:

  1. Ingesting text into a searchable vector knowledge base (DynamoDB rag-chunks)
  2. Embedding documents using all-MiniLM-L6-v2 via AutoTokenizer/AutoModel
  3. On user query, performing cosine similarity retrieval of top-k results
  4. Traversing a relational knowledge graph (NetworkX on rag-graph DynamoDB table)
  5. Merging vector context and graph context into a unified system prompt
  6. Routing the enriched prompt through Groq (llama-3.1-8b-instant)
  7. Persisting every conversation turn to DynamoDB rag-memory by session_id
  8. On future queries from the same session, loading history to maintain continuity

THE ORCHESTRATOR ADDS:

  A multi-agent pipeline where incoming tasks are:
  1. Decomposed by MODERATOR agent into RETRIEVAL, OUTPUT, VALIDATION sections
  2. Researched by KNOWLEDGE RETRIEVAL agent using live RAG and graph context
  3. Executed by CODER agent against platform-validated specifications
  4. Gated by SYSTEM DEBUG agent before any output is delivered to the user

IMMEDIATE STATE AS OF THIS REPORT:
Phases 1 through 5 are complete and operational. The full stack is live on AWS
free tier. All five Lambda functions are deployed and confirmed tested.

LONG-TERM GOAL:
Phases 6 through 8 will add multi-modal inputs, science and engineering domain
intelligence, and automated product generation from raw ideas.

SECTION 2: LONG-TERM SYSTEM ARCHITECTURE VISION
================================================================================

The architecture decisions made in every phase have been made with the Phase 8
final system in mind. Nothing is throwaway. Every component is permanent and
load-bearing.

PHASE 1 — COMPLETE: RAG PIPELINE
  Basic ingest, retrieve, and generate pipeline. all-MiniLM-L6-v2 embedded
  offline in Lambda container. DynamoDB vector storage. Groq LLM generation.

PHASE 2 — COMPLETE: END-TO-END INTEGRATION
  All three core Lambda functions operational. HuggingFace Space frontend live.
  DynamoDB clean with production records only.

PHASE 3 — COMPLETE: CONVERSATION MEMORY PERSISTENCE
  DynamoDB rag-memory table stores and retrieves conversation history by
  session_id. get_history uses query (not get_item) to handle composite key.
  save_history writes session_id, timestamp, turns, and updated fields.
  Memory confirmed persistent across separate messages in same session.

PHASE 4 — COMPLETE: GNN RELATIONAL KNOWLEDGE GRAPH
  NetworkX DiGraph library integrated into Lambda container. rag-graph DynamoDB
  table stores nodes and edges. rag-graph Lambda supports add_node and query
  actions. Graph traversal returns related nodes up to configurable depth.
  rag-chat merges graph context with RAG context in every Groq prompt.

PHASE 5 — COMPLETE: MULTI-AGENT ORCHESTRATION
  rag-orchestrator Lambda implements the full V3R agent pipeline. MODERATOR
  decomposes tasks. KNOWLEDGE RETRIEVAL queries live RAG and graph. CODER
  generates output. SYSTEM DEBUG gates all output. All agents operate via
  Groq prompt simulation on free tier.

PHASE 6 — PENDING: MULTI-MODAL INPUTS
  Image, diagram, and PDF input capability. Base64 ingest path into
  lambda_ingest_v4.py. Content extraction and embedding into rag-chunks.

PHASE 7 — PENDING: SCIENCE AND ENGINEERING DOMAIN INTELLIGENCE
  Domain-specific knowledge loading. Engineering ontologies. Scientific
  reasoning chains. Specialized retrieval for technical documentation.

PHASE 8 — PENDING: AUTOMATED PRODUCT GENERATION
  Accepts raw ideas as input. Outputs fully engineered product specifications.
  All prior phases are non-negotiable load-bearing infrastructure for this goal.

ARCHITECTURAL CHOICES AND RATIONALE:

  AWS Lambda: Scales without infrastructure management. Free tier covers all
  current and near-term usage. Supports container images up to 10 GB.

  DynamoDB: Scales to petabyte scale. PAY_PER_REQUEST is permanently free at
  current usage levels. No schema migration required as the system evolves.

  NetworkX: Pure Python graph library. No GPU required. No C extensions.
  Compatible with Lambda container environment. Sufficient for Phase 4 goals.
  Full GNN inference deferred to a phase where compute constraints are lifted.

  Groq / llama-3.1-8b-instant: Free API tier. Fast inference. Adequate for
  current pipeline. Model name must be verified with KNOWLEDGE RETRIEVAL before
  any hardcoded reference — deprecation risk is a registered forbidden pattern.

  AutoTokenizer + AutoModel (transformers): Required pattern for offline Lambda
  embedding. SentenceTransformer() direct instantiation is permanently forbidden
  due to incompatibility with the offline container environment.

  all-MiniLM-L6-v2: Lightweight, accurate, and fully compatible with the
  offline Lambda container. Loaded at /var/task/models/all-MiniLM-L6-v2.
  Model is never downloaded at runtime — baked into the container image.

SECTION 6: SYSTEM ARCHITECTURE — CURRENT BUILD
================================================================================

USER INTERFACE:
  HuggingFace Space (dnsxt/rag-memory) → Gradio 6.x UI
  Two tabs: Chat (sends to CHAT Lambda URL) and Ingest (sends to INGEST Lambda URL)

INGEST PIPELINE:
  HF Space Ingest Tab
  → rag-ingest-ultra-test Lambda
  → Chunks text at 500 words / 50 word overlap
  → Embeds each chunk via AutoTokenizer + AutoModel (all-MiniLM-L6-v2)
  → Stores to DynamoDB rag-chunks with uuid, source, timestamp, embedding

RETRIEVAL PIPELINE:
  rag-retrieve Lambda
  → Full DynamoDB rag-chunks scan
  → Cosine similarity scoring against query embedding
  → Returns top-k results (default 3)

CHAT PIPELINE:
  HF Space Chat Tab
  → rag-chat Lambda
  → get_history(session_id) — queries rag-memory table by session_id
  → rag-retrieve invoked for vector context (if use_rag=True)
  → rag-graph invoked for relational context (v3r_platform depth 2)
  → Merges history + RAG context + graph context into Groq system prompt
  → Groq llama-3.1-8b-instant generates response
  → save_history(session_id, turns) — writes to rag-memory with timestamp
  → Returns answer, rag_used flag, memory_turns count

GRAPH PIPELINE:
  rag-graph Lambda
  → Scans rag-graph DynamoDB table
  → Builds NetworkX DiGraph in memory
  → query action: returns related nodes up to specified depth
  → add_node action: writes new node with edges to DynamoDB

ORCHESTRATION PIPELINE:
  rag-orchestrator Lambda
  → get_rag_context(task) — invokes rag-retrieve
  → get_graph_context() — invokes rag-graph
  → MODERATOR agent (Groq): decomposes task into RETRIEVAL/OUTPUT/VALIDATION
  → KNOWLEDGE RETRIEVAL agent (Groq): surfaces validated platform-specific info
  → CODER agent (Groq): generates platform-compatible code
  → SYSTEM DEBUG agent (Groq): gates output — CLEARED or CORRECTIONS REQUIRED
  → Returns plan, knowledge, output fields
