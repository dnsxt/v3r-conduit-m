--------------------------------------------------------------------------------
BUG 1 — WRONG PARTITION KEY IN get_history() — CRITICAL
--------------------------------------------------------------------------------
Location: Line 21
Symptom: get_history always returned empty — conversation history never loaded
Diagnosed via: CloudWatch log — "Query condition missed key schema element: session_id"

Code before fix:
  KeyConditionExpression="vector_id = :sid",

Code after fix:
  KeyConditionExpression="session_id = :sid",

ROOT CAUSE:
  The rag-memory DynamoDB table partition key is session_id, not vector_id.
  The handoff report from the previous session incorrectly documented the PK
  as vector_id. This caused every get_history query to be rejected by DynamoDB.
  History was never loaded, so the Groq prompt always had zero conversation
  context, and session persistence was completely non-functional.

RESULT: Fixed. Memory persistence confirmed working after correction.

--------------------------------------------------------------------------------
BUG 2 — WRONG PARTITION KEY IN save_history() — CRITICAL
--------------------------------------------------------------------------------
Location: Line 35
Symptom: save_history put_item rejected — conversation turns never written
Diagnosed via: CloudWatch log — "Missing the key session_id in the item"

Code before fix:
  "vector_id": session_id,

Code after fix:
  "session_id": session_id,

ROOT CAUSE:
  Same incorrect key name as Bug 1. Both get_history and save_history were
  using vector_id throughout, causing complete read and write failure on
  the rag-memory table.

NOTE ON LINE NUMBERS:
  V3R initially reported these edits as lines 22 and 25. David corrected:
  actual lines were 21 and 35. Handler Reference Log corrected accordingly.
  This discrepancy arose from approximating line numbers from photos rather
  than reading from Notepad++ line counter directly.

RESULT: Fixed. DynamoDB put_item confirmed succeeding after correction.

--------------------------------------------------------------------------------
BUG 3 — GROQ PAYLOAD NOT ENCODED TO BYTES — MEDIUM
--------------------------------------------------------------------------------
Location: Line 83
Symptom: Groq returning empty response (masked by dead key — see Section 4)
Diagnosed via: Visual inspection of lambda_chat.py photo (Image 3 of 3)

Code before fix:
  conn.request("POST", "/openai/v1/chat/completions", body=payload,
  headers={"Authorization": f"Bearer {GROQ_API_KEY}",
  "Content-Type": "application/json"})

Code after fix:
  conn.request("POST", "/openai/v1/chat/completions",
  body=payload.encode("utf-8"),
  headers={"Authorization": f"Bearer {GROQ_API_KEY}",
  "Content-Type": "application/json",
  "Content-Length": str(len(payload.encode("utf-8")))})

ROOT CAUSE:
  http.client.HTTPSConnection requires the request body to be bytes, not a
  string. Additionally, the Content-Length header was missing entirely.
  Without explicit Content-Length, some HTTP servers may reject the request
  or fail to parse the body correctly.

RESULT: Fix applied. Final resolution of Groq "No response" was the dead API
key (Section 4), but this encoding fix is correct and load-bearing. It prevents
a class of silent failures that would surface with any future Groq model or
endpoint change.

PURPOSE:
  Manages the V3R relational knowledge graph. Stores nodes and edges in
  DynamoDB rag-graph table. Builds NetworkX DiGraph in memory on each
  invocation. Supports node addition and relationship traversal queries.

CREATION METHOD:
  Written in Notepad++ and saved to ecr-build directory before rebuild.
  Included in container image on first rebuild after Phase 4 initiation.

DEPLOYMENT ISSUE — RESOLVED IMMEDIATELY:
  First test of rag-graph Lambda returned:
  "Unable to import module 'lambda_graph': No module named 'lambda_graph'"

  ROOT CAUSE: lambda_graph.py was created AFTER the Phase 4-5 rebuild
  had already completed. The file existed on disk but was not inside the
  container image.

  RESOLUTION: Second rebuild executed. lambda_graph.py included in image.
  Function redeployed. Test passed immediately after second rebuild.

FUNCTIONS IMPLEMENTED:
  load_graph()
    Performs full DynamoDB scan of rag-graph table.
    Constructs NetworkX DiGraph from stored nodes and edges.
    Each node carries label and node_type attributes.
    Each edge carries relation attribute.

  query_related(node_id, depth)
    Calls load_graph() to build fresh graph.
    Uses nx.descendants() to find all reachable nodes.
    Filters by nx.shortest_path_length() <= depth.
    Returns sorted list of related nodes by depth ascending.

  add_node(node_id, label, node_type, edges, source_chunk_id)
    Writes node record to DynamoDB rag-graph table.
    Edges stored as list of dicts with target and relation fields.

  lambda_handler(event, context)
    Routes on event["action"]:
    "query" → calls query_related, returns related array
    "add_node" → calls add_node, returns confirmation
    Unknown action → returns statusCode 400

TEST RESULTS:
  add_node test (v3r_platform with 3 edges): statusCode 200 confirmed
  query test (v3r_platform depth 2): returned rag_retrieve, rag_memory,
  rag_chat in depth order — CONFIRMED OPERATIONAL

SECTION 6: LAMBDA_ORCHESTRATOR.PY — NEW FILE CREATION
================================================================================

File: lambda_orchestrator.py
Lines: 58
Status: OPERATIONAL

PURPOSE:
  Implements the full V3R multi-agent orchestration pipeline in a single
  Lambda function. Four sequential Groq calls simulate MODERATOR, KNOWLEDGE
  RETRIEVAL, CODER, and SYSTEM DEBUG agents. Each agent receives a specialized
  system prompt engineered to match its simulation target's behavioral profile.

CREATION METHOD:
  Written in Notepad++ and saved to ecr-build directory. Included in same
  rebuild as lambda_graph.py (second rebuild of Phase 4/5 sequence).

FUNCTIONS IMPLEMENTED:
  groq_call(system_prompt, user_content, max_tokens)
    Core Groq HTTP call used by all four agents.
    Uses http.client.HTTPSConnection to api.groq.com.
    Payload encoded to utf-8. Content-Length header set explicitly.
    Returns content string from choices[0].message.content.
    Default fallback: "No response"

  moderator(task)
    Simulates GPT-4o orchestration behavior.
    System prompt: terse task decomposition into RETRIEVAL, OUTPUT,
    VALIDATION sections. Suppresses generalist hedging.
    Temperature: 0.3 (deterministic)

  knowledge_retrieval(retrieval_task, rag_context, graph_context)
    Simulates Perplexity AI retrieval behavior.
    System prompt: platform-constraint-aware, no speculation, flags
    uncertain information explicitly.
    Receives live RAG and graph context as input.

  coder(output_task, kr_output)
    Simulates DeepSeek Coder V2 behavior.
    System prompt: all Forbidden Patterns Registry entries embedded.
    Lambda-specific constraints explicit.
    max_tokens: 2048 (larger output window)
    Known weakness: generates generic boilerplate — system prompt
    tuning planned for Phase 5 refinement.

  debug_gate(coder_output, validation_task)
    Simulates OpenAI o3 validation behavior.
    System prompt: checklist-based review against validation criteria.
    Output must begin with CLEARED FOR DELIVERY or CORRECTIONS REQUIRED.
    max_tokens: 2048

  get_rag_context(query)
    Invokes rag-retrieve Lambda with query and top_k 3.
    Extracts text from results array.
    Joins with double newline.
    Silent pass on exception.

  get_graph_context(node_id, depth)
    Invokes rag-graph Lambda with query action.
    Extracts related node labels.
    Formats as "Related components: ..." string.
    Silent pass on exception.

  lambda_handler(event, context)
    Validates task field present.
    Calls get_rag_context and get_graph_context.
    Calls moderator to produce plan.
    Parses plan into RETRIEVAL, OUTPUT, VALIDATION sections by line scan.
    Calls knowledge_retrieval with RETRIEVAL section.
    Calls coder with OUTPUT section and KR output.
    Calls debug_gate with coder output and VALIDATION section.
    Returns: plan, knowledge, output fields.

DEPLOYMENT:
  Lambda created with 1024 MB memory and 300 second timeout.
  CMD override set to lambda_orchestrator.lambda_handler.
  GROQ_API_KEY environment variable set.

TEST RESULT:
  Input: {"task": "What is V3R and what are its components?"}
  Output: All three fields returned (plan, knowledge, output).
  MODERATOR: Produced valid RETRIEVAL/OUTPUT/VALIDATION decomposition.
  KNOWLEDGE RETRIEVAL: Accurately described V3R stack from live RAG
  and graph context — cloud-native, Lambda, DynamoDB, Groq, all-MiniLM-L6-v2.
  CODER: Generated generic class-based Python — not Lambda-specific.
  DEBUG GATE: Returned CORRECTIONS REQUIRED. Caught placeholder region

KNOWN ISSUE:
