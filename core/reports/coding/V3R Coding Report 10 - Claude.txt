SECTION 2: DOCKER DISK CLEANUP — DIAGNOSIS AND RESOLUTION
================================================================================

SYMPTOM: C:\ drive showed only 17.7 GB free of 237 GB on a machine with
minimal software installed.

DIAGNOSIS SEQUENCE:
  docker system df -v revealed:
    - Production image ml-offline-torch26: 12 GB
    - Additional stale images (minimal, bare-v1, v4-clean): ~17 GB combined
    - buildx_buildkit_lucid_kare0_state volume: 71.28 GB
    - Build cache: 32.96 GB
    - Total Docker footprint: ~133 GB

ROOT CAUSE:
  Docker Desktop on Windows stores all image layers, build cache, and
  BuildKit volumes inside a virtual disk file (docker_data.vhdx). Even
  after deleting images and cache, the .vhdx file does not shrink
  automatically — it retains its peak allocated size until explicitly
  compacted.

RESOLUTION SEQUENCE:
  1. docker image prune -a — removed stale images (41.91 kB reclaimed
     from layer references — actual data was inside vhdx)
  2. docker builder prune -af — removed 32.96 GB build cache
  3. docker stop [buildkit container ID] — stopped orphaned buildkit daemon
  4. docker rm [buildkit container ID] — removed container
  5. docker volume rm buildx_buildkit_lucid_kare0_state — removed 71 GB volume
  6. Optimize-VHD attempted — FAILED (requires Hyper-V, not available on
     Windows 10 Home)
  7. diskpart → select vdisk → compact vdisk — SUCCEEDED
     docker_data.vhdx compacted from 119.45 GB to ~4 GB
  8. C:\ free space recovered from 17.7 GB to 133 GB

LESSONS RECORDED:
  - Docker Desktop vhdx does not auto-shrink — must compact manually after
    large cleanup operations
  - Optimize-VHD requires Hyper-V (Windows 10 Pro/Enterprise only)
  - diskpart compact vdisk is the correct tool on Windows 10 Home
  - Docker buildkit volumes can grow to 70+ GB during multi-attempt builds
    and must be explicitly removed

NEW FORBIDDEN PATTERN:
  Assuming Docker disk cleanup commands free Windows disk space immediately.
  Always follow with diskpart compact vdisk after major Docker pruning.

SECTION 3: PLATFORM ARCHITECTURE FIX — linux/amd64 BUILD FLAG
================================================================================

SYMPTOM: aws lambda update-function-code returned:
  InvalidParameterValueException: The image manifest, config or layer media
  type for the source image is not supported.

ROOT CAUSE:
  Docker Desktop on Windows defaults to the host architecture (x86_64 Windows)
  rather than linux/amd64 when no --platform flag is specified. Lambda requires
  linux/amd64 exclusively.

FIX:
  All subsequent builds use:
  docker buildx build --platform linux/amd64 --provenance false --sbom false

  --provenance false and --sbom false flags added to suppress Docker BuildKit
  attestation manifests which can cause ECR manifest type incompatibility.

ADDED TO ALL BUILD COMMANDS PERMANENTLY. See Partition 7 PowerShell reference.

Lines before: 60

Status: OPERATIONAL

CHANGES APPLIED:

Addition 1 — New imports:
  import base64
  import io
  from pypdf import PdfReader

Addition 2 — embed() normalization added:
  embedding = torch.nn.functional.normalize(embedding, p=2, dim=1)

  RATIONALE: all-MiniLM-L6-v2 was trained and benchmarked with normalized
  outputs. HuggingFace sentence-transformers normalizes by default. Without
  normalization, embedding magnitude varies with input length, introducing
  length bias into cosine similarity scoring. Normalization ensures all
  vectors lie on a unit hypersphere — cosine similarity reduces to dot
  product, improving retrieval accuracy and numerical stability.

  CONSEQUENCE: All 7 existing rag-chunks records stored without normalization
  required deletion and re-ingest. See Section 7.

Addition 3 — extract_pdf_text() function:
  Uses pypdf PdfReader with io.BytesIO — no temp file required.
  Extracts text from all pages, strips whitespace, joins with double newline.
  Returns empty string if no extractable text found.

Addition 4 — lambda_handler PDF routing:
  Added pdf_base64 branch before existing text branch.
  If pdf_base64 present: decode base64 → extract PDF text → chunk → embed
  If text present: existing behavior unchanged
  If neither: returns statusCode 400 with descriptive error
  source field: uses body.get("source", "manual") — unchanged behavior

CONFLICT AVOIDED:
  V3R initially proposed a replacement with different variable names
  (dynamodb instead of db) and a different embed() signature including
  max_length=512 parameter and normalization in a different position.
  Visual audit of existing file caught all conflicts before replacement.
  Final replacement preserves original variable names and embed() structure
  with only the normalization line and PDF additions applied.
