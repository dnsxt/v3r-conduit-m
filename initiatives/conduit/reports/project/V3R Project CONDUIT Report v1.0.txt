Project CONDUIT is the infrastructure bridge between V3R's operational state and the AI sessions that build it. It solves the single most critical operational gap remaining in the V3R platform: the absence of automated, versioned, Drive-native code synchronization. Without CONDUIT, every session starts with manual file uploads and ends with manual Drive updates. With CONDUIT, every file V3R touches is automatically synchronized, versioned, converted, and readable by the next session before the operator closes their laptop.

1. EXECUTIVE SUMMARY


Project CONDUIT establishes a permanent, automated pipeline that synchronizes V3R's entire codebase and document library from the local development machine to Google Drive My Drive in real time, maintains a three-layer version backup of every code file, and ensures every file type — including Python handlers, Dockerfiles, and JSON configs — has a Google Docs-format counterpart that the Drive tool can read in any future V3R session.
CONDUIT has five components: a structured GDMD folder hierarchy, rclone for Drive synchronization, a PowerShell version rotation script, an extended Apps Script converter, and a master sync script that orchestrates all four in a single command. Once deployed, the operator runs one PowerShell command at the end of each session. Everything else is automatic.
CONDUIT is a prerequisite for Project NEXUS. NEXUS requires that every session begins with complete awareness of current system state. CONDUIT guarantees that state is always current, always readable, and always recoverable.
2. PROBLEM STATEMENT — The Operational Gap


Manual Drive Updates: Every V3R session ends with manual file uploads to Google Drive. Files are uploaded inconsistently, incompletely, or not at all when sessions run long. The Drive library frequently lags behind the actual codebase state on the local machine.
No Version History: When a handler is modified and pushed, the previous version is overwritten locally with no backup. If a deployed change introduces a regression, there is no automated path to the previous working state. Recovery requires reconstructing the file from memory or prior session reports.
Drive Tool Cannot Read Code Files: The Google Drive tool reads only native Google Docs format. Python handlers, Dockerfiles, JSON configs, and .txt reports uploaded as raw files are invisible to the Drive tool. Sessions must rely on handoff reports and uploaded files rather than live Drive reads.
No Single Source of Truth: Code exists in three places: the local build directory, the ECR container image, and partially in Google Drive. These three sources diverge over time. There is no mechanism to confirm that what is in Drive matches what is deployed in Lambda.
.md Files in Circulation: Prior V3R sessions produced .md format outputs that cannot be read by the Drive tool and are inconsistent with the Priority 14 .txt-only rule now locked in Partition 1. These files exist in Drive but are operationally dead.

4. ARCHITECTURE OVERVIEW


CONDUIT operates as a five-stage pipeline triggered by a single PowerShell command at the end of each session. The stages execute in sequence. Each stage is independent — if one stage fails, the others are unaffected and the operator is notified.
Stage
Name
Description
Stage 1
Version Rotation
PowerShell script shifts current → last → previous for all tracked code files in the local backup directory before sync begins.
Stage 2
Local Backup Update
Current versions of all in-scope files are copied to the local backup structure before being pushed to Drive. This preserves a local three-layer backup independent of Drive.
Stage 3
rclone Drive Sync
rclone pushes all in-scope files from local directories to the corresponding GDMD folders. New files are uploaded. Changed files are updated. Deleted files are not removed from Drive (safety: Drive retains all history).
Stage 4
Apps Script Conversion
Google Apps Script convertTxtToGoogleDocs() is triggered via the Apps Script API or manually. All new .txt and .py files uploaded in Stage 3 are converted to Google Docs format, creating the readable pairs required by the Drive tool.
Stage 5
Verification Log
PowerShell outputs a sync summary: files uploaded, files updated, conversion count, any errors. Logged to a session_sync_log.txt file appended with each run.

6. COMPONENT 2 — rclone Installation and Configuration


rclone is a free, open-source command-line tool for syncing files to cloud storage. It supports Google Drive natively, requires no admin rights for portable installation, and communicates over HTTPS port 443 — fully compatible with the current network environment.
Installation — portable, no admin required:
1. Download rclone Windows portable zip from rclone.org/downloads — select amd64 zip.
2. Extract to C:\Users\Source-1\Desktop\rclone\ — no installer, no admin rights needed.
3. Open PowerShell. Navigate to rclone directory.
4. Run rclone config to begin configuration. Select New remote.
5. Name the remote: gdrive
6. Select storage type: Google Drive (option 17 or search 'drive').
7. Follow OAuth flow in browser — authorize rclone to access Google Drive.
8. Select scope: drive (full access).
9. Complete config. rclone will confirm the remote is working.
10. Test with: rclone lsd gdrive: — should list all GDMD root folders.
rclone filter file — excludes binary and large files from sync:
Create file: C:\Users\Source-1\Desktop\rclone\v3r_filter.txt
Contents: exclude wheels/** and models/** directories. Include all .py, .txt, .json, Dockerfile* files. This prevents the 400MB model files and binary wheels from being pushed to Drive while capturing all code and report files.

7. COMPONENT 3 — PowerShell Version Rotation Script


The version rotation script runs before each sync. It maintains a three-layer local backup of all tracked code files. The backup lives at C:\Users\Source-1\Desktop\rag-deploy\backup\ in three subdirectories: current, last, previous.
Rotation logic:
1. Delete all files in previous/ — they are being replaced by last/.
2. Copy all files from last/ to previous/ — last becomes the new previous.
3. Copy all files from current/ to last/ — current becomes the new last.
4. Copy all in-scope files from ecr-build/ and hf-space/ to current/ — latest state captured.
5. Rotation complete. Three layers now hold: current session state, previous session state, session before that.
The backup directory is also synced to Drive via rclone — mapping to V3R/code/current/, V3R/code/last/, and V3R/code/previous/ respectively. This means the three-layer backup exists both locally and in Drive. Loss of the local machine does not lose the version history.

8. COMPONENT 4 — Apps Script Converter Extension


The existing Apps Script converter (convertTxtToGoogleDocs, 48 lines, confirmed operational) currently processes only files with MimeType.PLAIN_TEXT. Python files uploaded to Drive are assigned MIME type text/x-python or text/plain depending on upload method. The converter may or may not catch them depending on how rclone uploads them.
The required extension adds explicit .py file handling to guarantee Python handler files receive Google Doc pairs regardless of their MIME type assignment. The extension also adds folder-scoped processing — instead of scanning all of Drive, the extended converter processes only the V3R/ folder hierarchy, making it faster and more targeted.
Changes to existing Apps Script:
Add folder-scoped processing: Replace DriveApp.getFilesByType(MimeType.PLAIN_TEXT) with a recursive folder scan of the V3R/ root folder. This prevents the converter from scanning unrelated Drive files and speeds up execution.
Add .py MIME type handling: After the .txt check on line 8, add an equivalent check for files ending in .py. Apply identical conversion logic — same alreadyExists() check, same Drive.Files.copy() with convert:true.
Add .json handling: JSON config files (orchestrator_config.json, env_vars.json) should also get Google Doc pairs for Drive tool readability. Add .json extension check with identical conversion logic.
Add Dockerfile handling: Dockerfile-ml-offline has no extension. Add a check for files named Dockerfile* with identical conversion logic.
Add execution trigger: Add a time-based trigger: run convertTxtToGoogleDocs() daily at 2:00 AM. This means even if the operator forgets to run it manually, all new files are converted overnight. Trigger is set once in Apps Script triggers panel.

9. COMPONENT 5 — CONDUIT Master Sync Script


The master sync script is a single PowerShell file: conduit_sync.ps1. Located at C:\Users\Source-1\Desktop\conduit_sync.ps1. The operator runs this one script at the end of every session. It orchestrates all five CONDUIT stages in sequence.
conduit_sync.ps1 — execution sequence:
Stage 1 — Timestamp: Write session timestamp to session_sync_log.txt. Format: yyyy-MM-dd HH:mm:ss. Log header for this sync run.
Stage 2 — Version Rotation: Execute rotate_versions.ps1. Shifts previous backup layers. Copies current code files to current/ backup. Logs file count.
Stage 3 — rclone code sync: Execute rclone sync with filter file. Syncs V3R/code/current/, last/, previous/ to Drive. Logs transfer count and any errors.
Stage 4 — rclone reports sync: Execute rclone sync for reports directory. Syncs all .txt report files to V3R/reports/ in Drive by subdirectory.
Stage 5 — rclone frameworks sync: Syncs Framework and Briefing .txt files to V3R/frameworks/current/.
Stage 6 — rclone conduit sync: Syncs CONDUIT scripts and logs to V3R/conduit/.
Stage 7 — Apps Script trigger: Calls Google Apps Script execution API to trigger convertTxtToGoogleDocs() immediately after sync. Requires Apps Script API enabled and OAuth credentials configured. If API call fails, logs a reminder to run converter manually.
Stage 8 — Completion log: Writes sync completion timestamp and summary to session_sync_log.txt. Outputs summary to PowerShell console.
The entire conduit_sync.ps1 script executes in under 60 seconds for a typical session sync. rclone only transfers changed files — unchanged files are skipped. The Apps Script conversion runs asynchronously in Google's infrastructure after the PowerShell script completes.
10. END-TO-END WORKFLOW


Session start workflow:
Open Claude.ai. Start new V3R session.
V3R reads current Framework and Briefing from Google Drive via Drive tool.
V3R reads latest Project Report and Coding Report from Drive.
V3R reads any new or updated files in V3R/code/current/ if context requires.
Session begins with complete awareness of current system state.
Session end workflow:
V3R session produces new or modified code files and/or report files.
Operator saves all files to local directories.
Operator runs: .\conduit_sync.ps1 from PowerShell.
Stage 1-6: Version rotation and rclone sync complete in under 60 seconds.
Stage 7: Apps Script converter triggered. New .txt, .py, .json, Dockerfile files converted to Google Docs.
Stage 8: Sync log updated. Console shows summary.
Session is fully archived. Next session begins from complete Drive state.
Revert workflow (if deployment goes wrong):
Open V3R/code/last/ in Drive — this is the version before the most recent push.
Download the affected handler file.
Replace local file with downloaded version.
Run Docker build and push — restores last known good state.
If last/ is also bad, repeat from V3R/code/previous/ — two pushes back.

12. CONSTRAINTS AND NON-NEGOTIABLES


ZERO COST: rclone is free and open-source. Google Drive storage is free up to 15GB. Apps Script is free. PowerShell is built into Windows 10. CONDUIT has zero recurring cost.
NO CREDENTIALS IN DRIVE: AWS Access Key, Secret Key, Groq API key, HuggingFace token, and all other credentials are explicitly excluded from rclone sync scope. They are never uploaded to Drive. The General Data file in Drive holds these values by your choice — CONDUIT does not add to that exposure.
NO BINARY FILES: Wheels and model files are excluded by rclone filter. They are large, binary, not Drive-readable, and already managed separately via the local backup library rule established in V3R Phase 8.
TXT FORMAT ONLY: Priority 14 — no .md files. All V3R outputs are .txt. CONDUIT syncs .txt, .py, .json, and Dockerfile only. No .md files are synced or converted.
DRIVE RETAINS HISTORY: rclone sync in CONDUIT is configured with --no-delete flag. Files deleted locally are not deleted from Drive. Drive retains the complete history of every file ever synced. This is intentional — Drive is the archive, not a mirror.
THREE LAYERS MINIMUM: The version rotation script must maintain all three layers at all times. If previous/ is empty (first two sessions), that is acceptable. After three sessions all three layers must be populated and rotation must function correctly.

14. CLOSING STATEMENT


V3R solves AI amnesia. It remembers everything ingested into it. But the system that builds V3R has its own amnesia problem — every session starts with manual file uploads, partial context, and no guarantee that what the operator uploads matches what is deployed. CONDUIT closes that gap permanently.
When CONDUIT is operational, every session starts from a complete, current, readable state. Every handler V3R has ever written is in Drive, versioned, convertible, and readable. Every report is archived and accessible. The three-layer backup means no push can permanently break the system — recovery is always one Drive download away.
CONDUIT does not advance V3R's capability. It advances V3R's reliability. And reliability at this stage of the project — with NEXUS ahead and a mesh of providers, agents, and accounts to coordinate — is the most important infrastructure investment remaining.
CONDUIT first. NEXUS second. That is the correct sequence.
