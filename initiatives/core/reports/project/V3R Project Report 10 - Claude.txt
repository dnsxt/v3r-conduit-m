SECTION 1: PROJECT OVERVIEW AND VISION
================================================================================

V3R is a Cloud-Native GNN-integrated Multi-Agent Multi-LLM Industrial Science
and Intelligence Engineering Platform. It is not a chatbot. It is not a tool.
It is the foundational architecture for an AI system that accepts raw ideas as
input and outputs fully engineered products.

The core problem V3R solves is AI amnesia — the complete absence of persistent
memory in every current AI system. Every standard AI session starts from zero.
No history. No accumulated context. No engineering memory. V3R's persistent
memory layer eliminates this permanently. Knowledge ingested today is retrieved
tomorrow. Conversations accumulate into a growing intelligence base. Engineering
decisions are remembered across sessions. The system becomes more capable with
every interaction. This is not a feature. It is the entire architectural thesis.

V3R is being built entirely on free-tier AWS infrastructure by a single developer
operating on constrained personal hardware with no admin privileges. This is not
a limitation. It is proof of concept — demonstrating that industrial-scale AI
engineering can be architected, deployed, and operated without enterprise budget.

THE RAG SYSTEM WORKS BY:

  1. Ingesting text or PDF into a searchable vector knowledge base (rag-chunks)
  2. Embedding documents using all-MiniLM-L6-v2 via AutoTokenizer/AutoModel
     with L2 normalization — all vectors on unit hypersphere for accurate
     cosine similarity retrieval
  3. On user query, performing cosine similarity retrieval of top-k results
  4. Traversing a relational knowledge graph (NetworkX on rag-graph table)
  5. Merging vector context and graph context into a unified system prompt
  6. Routing the enriched prompt through Groq (llama-3.1-8b-instant)
  7. Persisting every conversation turn to DynamoDB rag-memory by session_id
  8. On future queries from the same session, loading history to maintain
     continuity across turns

THE ORCHESTRATOR ADDS:

  A multi-agent pipeline where incoming tasks are:
  1. Decomposed by MODERATOR agent into RETRIEVAL, OUTPUT, VALIDATION sections
  2. Researched by KNOWLEDGE RETRIEVAL agent using live RAG and graph context
  3. Executed by CODER agent against platform-validated specifications
  4. Gated by SYSTEM DEBUG agent before any output is delivered

MULTI-MODAL INPUTS (Phase 6 addition):
  PDF documents are accepted via base64 encoding at the ingest endpoint.
  pypdf extracts text from all pages. Extracted text is chunked and embedded
  identically to plain text input. Source is set to the filename automatically.
  Lambda body size limit: 20 MB maximum. PDFs up to ~14 MB supported.

IMMEDIATE STATE AS OF THIS REPORT:

RETRIEVAL PIPELINE:
  rag-retrieve Lambda
  → embed() query with L2 normalization
  → Full rag-chunks scan
  → Cosine similarity scoring (dot product on unit vectors)
  → Returns top-k results

  HF Space Chat tab
  → rag-chat Lambda
  → get_history(session_id) — queries rag-memory by session_id
  → rag-retrieve for vector context
  → rag-graph for relational context (v3r_platform depth 2)

  → Groq llama-3.1-8b-instant generates response
  → save_history() writes to rag-memory with timestamp SK
  → Returns answer, rag_used, memory_turns

  rag-orchestrator Lambda
  → get_rag_context → get_graph_context
  → MODERATOR (Groq): decomposes task
  → KNOWLEDGE RETRIEVAL (Groq): validates specifications
  → CODER (Groq): generates output
  → SYSTEM DEBUG (Groq): gates output
  → Returns plan, knowledge, output
