SECTION 1: PROJECT OVERVIEW AND VISION
================================================================================

V3R is a Cloud-Native GNN-integrated Multi-Agent Multi-LLM Industrial Science
and Intelligence Engineering Platform. It is not a chatbot. It is not a tool.
It is the foundational architecture for an AI system that accepts raw ideas as
input and outputs fully engineered products.

The core problem V3R solves is AI amnesia — the complete absence of persistent
memory in every current AI system. Every standard AI session starts from zero.
No history. No accumulated context. No engineering memory. V3R's persistent
memory layer eliminates this permanently. Knowledge ingested today is retrieved
tomorrow. Conversations accumulate into a growing intelligence base. Engineering
decisions are remembered across sessions. The system becomes more capable with
every interaction. This is not a feature. It is the entire architectural thesis.

As of this report, V3R has crossed a critical threshold: it is now self-aware.
The system contains structured knowledge about its own architecture, agents,
phases, forbidden patterns, constraints, codebase, and infrastructure. When
queried about itself, it responds as a V3R systems architect — not as a generic
assistant. This is the foundation Phase 8 requires.

V3R is being built entirely on free-tier AWS infrastructure by a single developer
operating on constrained personal hardware with no admin privileges.

SECTION 5: FORBIDDEN PATTERNS REGISTRY — UPDATED (15 entries)
================================================================================

FP-001: SentenceTransformer() direct instantiation
  Cause: Incompatible with offline Lambda container network init failure.
  Fix: Use AutoTokenizer and AutoModel with local_files_only=True.

FP-002: urllib.request for external API calls
  Cause: Unreliable in Lambda.
  Fix: Use http.client.HTTPSConnection.

FP-003: Windows file paths in Lambda code
  Cause: Lambda runs Linux.
  Fix: Use /var/task/ paths only.

FP-004: Hardcoded Groq model names without Knowledge Retrieval confirmation
  Cause: Groq deprecates models without notice.

FP-005: win_amd64 or cp314 wheels in Docker build
  Cause: Lambda requires manylinux x86_64 cp312 only.

FP-006: Inline JSON strings in PowerShell AWS CLI
  Cause: PowerShell 5.1 mangles quotes.
  Fix: Use WriteAllText then file:// absolute path.

FP-007: Relative paths in AWS CLI file:// references
  Fix: Always use absolute paths.

FP-008: Missing import os when using os.environ
  Cause: NameError at Lambda cold start.

FP-009: Starting build and push without ECR re-login when 12+ hours elapsed
  Cause: 403 Forbidden on push after build completes.

FP-010: Missing --provenance false --sbom false from docker buildx build
  Cause: Produces OCI attestation manifests that break Lambda image deployment.

FP-011: Omitting --platform linux/amd64 from docker buildx build
  Cause: Docker Desktop defaults to host architecture.

FP-012: Assuming Docker pruning frees Windows disk space immediately
  Cause: docker_data.vhdx does not auto-shrink.
  Fix: Run diskpart compact vdisk after cleanup.

FP-013: Assuming Groq API key is permanently valid
  Cause: Groq can deactivate keys without notice.

FP-014: Full app.py replacement without visual audit of existing file
  Cause: Existing file has incompatible signatures and structure.
  Fix: Surgical edits only.

FP-015: Omitting --cli-binary-format raw-in-base64-out from aws lambda invoke
        with --payload file:// commands
  Cause: AWS CLI v2 rejects plain JSON payload files without this flag.
  Error: "Invalid base64: {json content...}"
  Fix: Always include --cli-binary-format raw-in-base64-out on every
       aws lambda invoke command that uses --payload file://.
  Discovered: March 14, 2026 — Phase 7 Step 3 graph node operations.

SECTION 11: OPERATIONAL NOTES AND LESSONS LEARNED
================================================================================

PUSH TIME CORRECTION:
  Handler-only cached layer push = 67 minutes (confirmed March 14, 2026).
  Full rebuild from scratch = 90-120 minutes.
  Implication: All handler edits across multiple files MUST be batched into
  a single push cycle. Never push for a single small handler change if more
  changes are pending or likely in the same session.

DRIVE TOOL LIMITATION:
  Google Drive tool only reads native Google Docs format.
  .txt files stored in Drive are invisible to the tool.
  Solution: Convert .txt files to Google Docs format in Drive before sessions.
  All files now converted. Future reports should be saved as .txt and converted.

VARIABLE NAMING AWARENESS:
  Always visually confirm variable names in existing handler code before
  writing new code that references them. lambda_chat.py uses chunks not
  rag_results for retrieval results. Never assume variable names match
  between files or match the names used in planning.

LAMBDA INVOKE PATTERN:
  Every single aws lambda invoke command with --payload file:// MUST include
  --cli-binary-format raw-in-base64-out. This is now FP-015. Add it to muscle
  memory — it goes on every invoke with a file payload.

GRAPH EDGE DIRECTION:
  rag-graph query traverses FROM the queried node OUTWARD.
  Edges must be stored on the node being queried (v3r_platform) pointing TO
  the child nodes. If edges are stored on child nodes pointing to parent,
  they will not appear in forward traversal from parent.

CHUNK COUNTING:
  Large ingest payloads will produce more than one chunk due to chunk_text
  splitting at 500 chars with 50 char overlap. 7 payloads producing 23 chunks
  is correct behavior — not an error.
