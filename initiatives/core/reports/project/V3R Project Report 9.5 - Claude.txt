## SECTION 1 — WHAT V3R IS

V3R is a Cloud-Native GNN-integrated Multi-Agent Multi-LLM Industrial Science and Intelligence Engineering Platform. It is not a chatbot. It is not a tool. It is the foundational architecture for an AI system that accepts raw ideas and outputs fully engineered products.

### The Problem It Solves

Every AI system in existence today starts each session from zero. No memory of prior conversations. No accumulated engineering context. No knowledge that persists. A developer using a standard AI assistant today gets the exact same blank-slate system they got six months ago, regardless of how much work has been done together.

V3R eliminates this permanently. The system is designed so that every piece of knowledge ingested, every conversation completed, and every engineering decision made accumulates into a persistent, retrievable, growing intelligence base. The longer V3R runs, the more capable it becomes. This is not a feature. It is the architectural thesis on which every component is built.

### The End Goal

Phase 8: V3R accepts a raw idea and outputs a fully engineered product. Not a draft. A product. Specified, architected, coded, tested, deployable. Every phase between now and Phase 8 is a non-negotiable prerequisite for that outcome.

### Who Is Building It

A single developer (Dirty) on a consumer-grade laptop with no admin access, no GPU, 7 Mbps upload speed, and a permanent AWS free-tier budget. This is intentional. When Phase 8 is reached, it will prove that industrial-scale AI engineering intelligence can be built without enterprise infrastructure.

## SECTION 2 — SYSTEM ARCHITECTURE

### Three-Layer Cognitive Model

```
LAYER 1 — MEMORY
Text → Lambda rag-ingest-ultra-test → Chunk → Embed → DynamoDB rag-chunks

LAYER 2 — REASONING
Query → Lambda rag-retrieve → Embed → Cosine Scan → Top-K Results

LAYER 3 — GENERATION
Query + Context + History → Lambda rag-chat → Groq API → Answer → Save History
```

### Live Infrastructure

| Component | Service | Identifier | Status |
|---|---|---|---|
| UI | HuggingFace Space | dnsxt/rag-memory | Running, public |
| Ingestion | AWS Lambda | rag-ingest-ultra-test | Operational |
| Retrieval | AWS Lambda | rag-retrieve | Operational |
| Generation | AWS Lambda | rag-chat | Partially broken (see Section 5) |
| Knowledge store | DynamoDB | rag-chunks | Clean, 6 records |
| Memory store | DynamoDB | rag-memory | Operational |
| Container registry | ECR | rag-ingest-container:ml-offline-torch26 | Single image, clean |

### Key Technical Decisions

**Offline embedding:** all-MiniLM-L6-v2 is baked into the Docker image. No runtime model downloads. No HuggingFace dependency in the inference path. Loaded via AutoTokenizer/AutoModel — never SentenceTransformer().

**Full table scan retrieval:** No vector database. DynamoDB full scan with in-Lambda cosine similarity. Correct for sub-10,000 record scale through Phase 4.

**Single shared image:** All three Lambda functions run from the same Docker image. Shared model eliminates redundant storage and build time.

**Free tier only:** AWS free tier is a permanent design constraint, not a temporary limitation.

## SECTION 5 — KNOWN ISSUES

### ISSUE-001 — rag-chat Returning "No Response"
**Severity:** High — core functionality impaired
**Status:** Diagnosis complete, resolution pending one verification step

**Symptom:** Lambda test returns statusCode 200 but body.answer = "No response" instead of Groq-generated text.

**Root cause:** Groq API returning empty choices array. Consistent with 401 authentication failure — GROQ_API_KEY environment variable is likely empty string in the currently deployed Lambda container.

**Evidence:** The response path `resp_data.get("choices", [{}])[0].get("message", {}).get("content", "No response")` falls through to the default "No response" value. A 401 Groq response contains an error object, not a choices array.

**Resolution — one step:**
```powershell
aws lambda get-function-configuration --function-name rag-chat --region us-east-1 --query "Environment.Variables"
```
If empty: re-set the key using WriteAllText method and update-function-configuration. No rebuild required.

**Impact on other components:** None. rag-ingest-ultra-test and rag-retrieve are fully operational and unaffected.
