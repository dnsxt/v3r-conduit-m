SECTION 1: PROJECT OVERVIEW AND VISION
================================================================================

This project is the construction of a cloud-native, serverless Retrieval-Augmented
Generation (RAG) system. The immediate purpose of this system is to give AI
assistants like Claude persistent memory and a personal knowledge base that
survives across individual conversation sessions.

The core problem being solved is well-documented: AI systems like Claude, ChatGPT,
and Grok have no persistent memory between conversations. Every new session starts
blank. This causes AI assistants to repeat themselves, lose context, contradict
prior advice, and generally behave inconsistently over time. The user experienced
this problem directly — both ChatGPT and Grok led the project in circles due to
this limitation, and this RAG system is the solution.

The RAG system works by:

  1. Ingesting documents (URLs, PDFs, text, notes) into a searchable knowledge base
  2. When a user sends a message, automatically searching the knowledge base for
     relevant stored information
  3. Injecting that retrieved information into the AI's context window before it
     generates a response
  4. Storing every conversation turn in a memory database so the AI can reference
     prior exchanges in future sessions

The result is an AI assistant that behaves consistently, remembers prior context,
does not repeat itself, and builds a personalized understanding of the user over time.

IMMEDIATE GOAL:
Get a minimum viable RAG system running as fast as possible so the user can begin
having consistent, memory-persistent AI conversations immediately.

LONG-TERM GOAL:
This minimum viable system is explicitly the foundation for a much larger system
described as a Cloud-Native GNN (Graph Neural Network) integrated with a
RAG-based Multi-Agent, Multi-Modal AI Science and Engineering Intelligence System.
The long-term system is intended to support an industrial-scale data cache including:
  - Product specifications
  - Quality control data
  - Systems designs
  - Workflow records
  - Design and reasoning records
  - Engineering documentation

The ultimate long-term goal is an AI system that accepts ideas as input and outputs
complete engineered products. This is a significant and ambitious research and
engineering platform, and the current RAG build is explicitly Step 1 of that journey.

SECTION 2: LONG-TERM SYSTEM ARCHITECTURE VISION
================================================================================

The user has described the long-term target as a Cloud-Native GNN integrated with
a RAG-based Multi-Agent Multi-Modal AI Science and Engineering Intelligence System.

Breaking this down:

GRAPH NEURAL NETWORK (GNN) LAYER:
  GNNs operate on graph-structured data — nodes and edges representing entities
  and their relationships. In the context of this system, the GNN would eventually
  model relationships between engineering concepts, products, processes, and data
  points. This goes beyond simple text retrieval and into structured knowledge
  representation. The RAG system being built now is a prerequisite — it stores and
  retrieves information. The GNN layer would add relational reasoning on top.

RAG LAYER (CURRENT BUILD):
  Retrieval-Augmented Generation is the foundational memory and retrieval system.
  Documents are chunked, embedded as vectors, stored, and retrieved by semantic
  similarity to user queries. This is what is being built right now.

MULTI-AGENT LAYER:
  Multiple specialized AI agents working in coordination. For example: a research
  agent, a design agent, a QC agent, a documentation agent — each with access to
  the shared knowledge base and each specialized for its domain.

MULTI-MODAL LAYER:
  Handling not just text but images, diagrams, CAD files, sensor data, and other
  non-text data types. This is a future expansion beyond the current text-only build.

SCIENCE AND ENGINEERING INTELLIGENCE:
  The system is ultimately positioned as a domain-specific AI for science and
  engineering work — product design, QC analysis, systems engineering, and
  eventually automated product generation from concept to specification.

The architecture decisions made in the current build have been made with this
long-term vision in mind. Specifically:
  - AWS Lambda was chosen because it scales infinitely without infrastructure management
  - DynamoDB was chosen because it scales to petabyte scale and is permanently free
    at the current usage level
  - The modular function design (separate ingest, retrieve, chat functions) allows
    each component to be upgraded independently as the system grows
  - The embedding model choice (sentence-transformers) can be upgraded to more
    powerful models later without changing the storage architecture

SECTION 7: SYSTEM ARCHITECTURE — CURRENT BUILD
================================================================================

DATA FLOW — INGESTION:
  User (any device/browser)
    → HuggingFace Space UI (Gradio app)
      → HTTP POST to rag-ingest Lambda Function URL
        → Lambda fetches/receives content
        → Chunks text into 512-char segments with 64-char overlap
        → Calls HuggingFace Inference API for embeddings
          (model: sentence-transformers/all-MiniLM-L6-v2)
        → Stores chunk + embedding + metadata to DynamoDB rag-chunks table
      → Returns success/failure status to UI
    → Status displayed to user

DATA FLOW — CHAT:
  User (any device/browser)
    → HuggingFace Space UI (Gradio app)
      → HTTP POST to rag-chat Lambda Function URL
        → Lambda loads conversation history from DynamoDB rag-memory
        → Lambda invokes rag-retrieve Lambda internally
          → rag-retrieve embeds query via HuggingFace API
          → Scans rag-chunks, computes cosine similarity
          → Returns top 5 most relevant chunks
        → Builds full prompt: system instructions + memory + RAG context + query
        → Calls Groq API (llama-3.3-70b-versatile)
        → Saves exchange to DynamoDB rag-memory
        → Returns response to UI
    → Response displayed to user in chat

EMBEDDING MODEL:
  sentence-transformers/all-MiniLM-L6-v2
  - Produces 384-dimensional dense vectors
  - Runs on HuggingFace Inference API (free tier)
  - Chosen for speed, small size, and strong semantic similarity performance
  - Compatible with future upgrade to larger models

SIMILARITY SEARCH:
  Cosine similarity computed in pure Python (no numpy)
  - Full table scan of rag-chunks
  - This is appropriate for small-to-medium knowledge bases (up to ~10,000 chunks)
  - Future upgrade path: DynamoDB Streams + OpenSearch, or migration to Pinecone/Weaviate

LLM INFERENCE:
  Groq API with llama-3.3-70b-versatile
  - Free tier: approximately 14,400 requests/day
  - Very fast inference (Groq uses custom LPU hardware)
  - 70B parameter model — high quality responses
  - Context window: 128,000 tokens

TEXT COMPRESSION:
  All stored text is zlib-compressed and base64-encoded before storage in DynamoDB.
  This significantly reduces storage consumption and DynamoDB read/write costs.
  Typical compression ratio for English text: 60-70% size reduction.

COST ANALYSIS:
  Lambda:     $0 (1M requests/month free, permanently)
  DynamoDB:   $0 (25 GB free, permanently — current usage is near zero)
  Groq:       $0 (free tier)
  HuggingFace Space: $0 (free CPU tier)
  HuggingFace Inference API: $0 (free tier, rate limited)
  Total monthly cost: $0

SECTION 10: CURRENT BLOCKING ISSUE — FULL ANALYSIS
================================================================================

SYMPTOM:
  When the user clicks "Ingest Text" or "Ingest URL" in the HuggingFace Space,
  the status field displays: "❌ Error: Unknown error"

  This error message is returned by the app.py code specifically when the Lambda
  returns an HTTP status code other than 200.

WHAT WE KNOW FOR CERTAIN:
  1. The rag-ingest Lambda IS deployed and running correctly.
     Evidence: AWS console direct test returned statusCode 200, "Ingested successfully"
  
  2. The HuggingFace Space IS running without errors.
     Evidence: Space shows "Running" status, Container logs show clean startup
  
  3. The Function URLs ARE created and configured.
     Evidence: Both URLs returned successfully from get-function-url-config
  
  4. The Lambda environment variables ARE set correctly.
     Evidence: get-function-configuration showed HF_API_KEY and DYNAMO_TABLE present
  
  5. The IAM permissions ARE correct.
     Evidence: All three required policies visible in IAM console
  
  6. The INGEST_LAMBDA_URL secret IS set in HF Space.
     Evidence: User confirmed it is set (contents cannot be viewed in console)

WHAT IS UNKNOWN:
  1. Whether the Lambda is actually being called from the HF Space
     (Network tab showed only a JWT token fetch, no POST to Lambda URL)
  
  2. Whether the HF Space secrets are being read correctly at runtime
  
  3. Whether the Lambda is returning 500 due to a runtime error when called
     via HTTP (as opposed to the direct invocation test format)

MOST LIKELY ROOT CAUSE:
  The Lambda is being called but failing at runtime with a 500 error specifically
  when invoked via HTTP Function URL (as opposed to direct Lambda invocation).
  
  The difference between the two invocation methods:
  
  DIRECT TEST (works):
    Event format: {"type":"text","content":"...","source":"..."}
    The event object IS the JSON directly
  
  HTTP FUNCTION URL CALL (fails):
    Event format: {"body": "{\"type\":\"text\",\"content\":\"...\",\"source\":\"...\"}", 
                   "headers": {...}, "requestContext": {...}, ...}
    The JSON is wrapped inside event["body"] as a string
  
  The code on line 101 of the Lambda handles this:
    body = json.loads(event.get("body","{}")) if isinstance(event.get("body"),str) else event
  
  This SHOULD correctly parse the HTTP-wrapped format. However there is a subtle
  bug possibility: if the HTTP request from HuggingFace sends the body in a format
  that the Lambda misinterprets, the type field could default to "url" instead of
  "text", causing the Lambda to try to fetch a URL with empty content and fail.

SECONDARY POSSIBLE CAUSE:
  The HuggingFace Inference API (used for embeddings) may be rate-limiting or
  returning errors when called from the Lambda. The free tier has strict rate limits.
  If the embedding call fails, the Lambda returns a 500 error. This would explain
  why the direct test (which uses a short test phrase) sometimes works but longer
  or repeated calls fail.

TERTIARY POSSIBLE CAUSE:
  The INGEST_LAMBDA_URL secret in the HF Space may have an invisible character,
  trailing space, or other formatting issue despite appearing correct. URL secrets
  are notoriously sensitive to whitespace.
