SECTION 1: PROJECT OVERVIEW AND VISION
================================================================================

V3R is a Cloud-Native GNN-integrated Multi-Agent Multi-LLM Industrial Science
and Intelligence Engineering Platform. It is not a chatbot. It is not a tool.
It is the foundational architecture for an AI system that accepts raw ideas as
input and outputs fully engineered products.

The core problem V3R solves is AI amnesia. Every standard AI session starts
from zero. No history. No accumulated context. No engineering memory. V3R's
persistent memory layer eliminates this permanently. Knowledge ingested today
is retrieved tomorrow. Conversations accumulate into a growing intelligence
base. Engineering decisions are remembered across sessions. The system becomes
more capable with every interaction. This is not a feature. It is the entire
architectural thesis.

As of this report V3R has crossed its most significant threshold: Phase 8 is
active and the Product Generator is operational. V3R now accepts a raw idea
as input and returns a fully structured multi-pass product specification. This
is the destination every prior phase was building toward.

V3R is being built entirely on free-tier AWS infrastructure by a single
developer operating on constrained personal hardware with no admin privileges.

SECTION 11: OPERATIONAL NOTES AND LESSONS LEARNED
================================================================================

PUSH SEQUENCING:
  lambda_ideagen.py must exist in build directory BEFORE docker build
  runs. File creation and push must be sequenced correctly. Create
  file first. Confirm file present in build directory. Then push.

FP-016 LESSON:
  Every Dockerfile COPY handler line must be visually verified for
  $(LAMBDA_TASK_ROOT) with $ before push executes. A missing $ is
  invisible at build time and only surfaces as RuntimeImportModuleError
  at Lambda cold start after a 67 minute push cycle.

WHEEL LIBRARY RULE (new this session):
  All packages built into the Docker image must exist as local wheel
  files in the wheels directory before build runs. No package is
  downloaded by Docker at build time. Prevents CCleaner or temp folder
  cleanup from destroying dependencies between sessions.

CMD OVERRIDE REQUIREMENT:
  Every Lambda function except rag-ingest-ultra-test requires a CMD
  override. Default CMD is lambda_ingest_v4.lambda_handler. Any
  function without a correct CMD override will execute the wrong handler.
  Verify CMD override on every newly created Lambda function immediately
  after creation.

HF SPACE VARIABLES vs SECRETS:
  CHAT_LAMBDA_URL and INGEST_LAMBDA_URL are currently in Variables
  (public). They should be in Secrets (private). Move at next opportunity.
