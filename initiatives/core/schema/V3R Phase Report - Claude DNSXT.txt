## PHASE 1 — RAG PIPELINE FOUNDATION

### Status: COMPLETE

### Objective
Build the foundational three-layer RAG pipeline: ingest text into a vector knowledge base, retrieve semantically relevant chunks by query, generate augmented responses using an LLM. Establish the offline-first Lambda container strategy and confirm all components function in isolation.

### Scope
- Docker image with offline embedding model
- DynamoDB table for vector storage
- Lambda function for ingestion
- Lambda function for retrieval
- Lambda function for LLM generation
- Groq API integration

### Key Technical Decisions Made in Phase 1

**AutoTokenizer/AutoModel over SentenceTransformer()**
SentenceTransformer() was attempted first. It performs network initialization checks even with local_files_only=True, causing failures in the offline Lambda container. AutoTokenizer.from_pretrained() and AutoModel.from_pretrained() with local_files_only=True load purely from disk. This was the first entry in the Forbidden Patterns Registry.

**Word-level chunking at 500/50**
Chunk size of 500 words with 50-word overlap established. Rationale: semantic coherence at 500 words, boundary context preservation at 50-word overlap. This became the permanent chunking strategy.

**Full DynamoDB scan for retrieval**
Vector similarity search via full table scan. No vector database. OpenSearch and Pinecone evaluated and rejected — both require paid tiers. Full scan is correct for sub-10,000 record scale through Phase 4.

**Cosine similarity as the scoring metric**
Dot product and Euclidean distance evaluated. Cosine chosen for magnitude-independence — semantic similarity is direction in embedding space, not magnitude.

### Failures and Resolutions in Phase 1

| Failure | Resolution |
|---|---|
| SentenceTransformer() network init failure in Lambda | Replaced with AutoTokenizer/AutoModel, added to Forbidden Patterns Registry |
| win_amd64 wheels in Docker build | Identified and purged from wheels/ directory, added to Forbidden Patterns Registry |
| urllib.request failures in Lambda HTTP calls | Replaced with http.client.HTTPSConnection, added to Forbidden Patterns Registry |

### Key Technical Decisions Made in Phase 2

**Single shared ECR image for all functions**
All three Lambda functions share the same Docker image. Individual function behavior is set by the Lambda console handler configuration. Rationale: shared model eliminates redundant storage and build time.

**HuggingFace Spaces over AWS frontend**
Free Gradio hosting with environment variable support. AWS S3+CloudFront and EC2 both require paid services beyond free tier. Spaces provides equivalent functionality at zero cost.

**Function URL over API Gateway**
API Gateway requires paid service for production use beyond free tier limits. Lambda Function URLs provide direct HTTPS invocation at no additional cost.

**PAY_PER_REQUEST billing on DynamoDB**
No provisioned capacity. PAY_PER_REQUEST means zero cost when idle, which is the dominant state for a development-stage system. Phase 3+ usage will remain within free tier WCU/RCU allocations.

## PHASE 3 — CONVERSATION MEMORY PERSISTENCE

### Status: ACTIVE
### Started: March 12, 2026
### Target Completion: March 2026

### Objective
Make rag-chat remember conversation history across turns within a session. When a user tells the system their name in turn 1, the system recalls it in turn 2. Conversation turns are persisted to rag-memory and retrieved on each subsequent turn. This is the first realization of V3R's core thesis — a system that accumulates context across interactions.

### Scope
- Fix get_history() in lambda_chat.py to use DynamoDB query instead of get_item
- Fix save_history() to include timestamp sort key
- Move GROQ_API_KEY from hardcoded string to Lambda environment variable
- Add import os to lambda_chat.py
- Rebuild and push Docker image
- Redeploy rag-chat
- Verify memory recall in HF Space

### Work Completed (March 12, 2026)

**Root Cause Identified**
rag-memory table has composite key: vector_id (S) PK + timestamp (N) SK.
Original get_history() used get_item with only vector_id — DynamoDB get_item requires all keys for composite key tables. Silent failure. History always returned empty.

**get_history() Fixed**
Replaced get_item with memory_table.query using KeyConditionExpression. Returns most recent record's turns list. Handles empty results gracefully.

**save_history() Fixed**
Added timestamp as integer unix epoch to satisfy sort key requirement on put_item.

**GROQ_API_KEY Secured**
Moved from hardcoded string to os.environ.get("GROQ_API_KEY", "").
Environment variable set in Lambda console via WriteAllText + update-function-configuration.
Confirmed set via get-function-configuration query.

**import os Added**
Added to line 2 of lambda_chat.py after NameError: os not defined was caught in Lambda test.

**Docker Builds Completed**
Build 1: Full rebuild ~2.5 hours — completed, push failed on ECR 403 (expired token), resolved by re-login and direct push.
Build 2: Cached rebuild — import os fix deployed.
Build 3: Cached rebuild — get_history/save_history fix deployed.

### Work Remaining

**Single remaining issue: Groq returning "No response"**

The Lambda test returns statusCode 200 but the body answer field contains "No response" instead of a Groq-generated answer. The error path that produces this is:

**Most likely cause:** GROQ_API_KEY environment variable is empty string in the currently deployed container. The variable was confirmed set once during the session but subsequent Lambda redeployments may have affected the configuration.

## PHASE 4 — GNN RELATIONAL KNOWLEDGE GRAPHS

### Status: PENDING

### Planned Objective
Layer a graph neural network on top of the existing vector knowledge base. Instead of returning flat ranked chunks, rag-retrieve returns a subgraph of related knowledge nodes. Relationships between concepts are encoded as graph edges. Multi-hop reasoning becomes possible — "what is related to X that is also related to Y?"

### Planned Scope
- Graph adjacency table in DynamoDB or evaluation of AWS Neptune free tier
- GNN model selection (PyTorch Geometric or DGL — Knowledge Retrieval Agent must confirm)
- New lambda function rag-graph or expanded rag-retrieve
- Node feature vectors: existing 384-dimensional embeddings become node features
- Edge construction: semantic similarity above threshold creates edges
- Graph traversal layer between retrieval and generation

### Architectural Prerequisites
- Phase 3 must be complete (conversation memory must be stable)
- Knowledge Retrieval Agent must confirm GNN library compatibility with Python 3.12 Lambda
- Infrastructure Agent must evaluate Neptune free tier availability
- Full rebuild will be required (new GNN library in Docker image)

### Known Risks
- GNN libraries (PyTorch Geometric) have complex wheel dependencies — Linux manylinux cp312 availability must be confirmed before planning the build
- Graph construction at scale may exceed Lambda memory limits — requires profiling
- Neptune is not on free tier — DynamoDB adjacency table approach is the free-tier-compliant fallback

## PHASE 5 — MULTI-AGENT ORCHESTRATION

### Status: PENDING

### Planned Objective
Route incoming queries to specialized agents based on intent classification. A coding question goes to a code-specialist agent. A retrieval question goes to a knowledge-specialist agent. A debugging request goes to a debug-specialist agent. Each agent has its own system prompt, its own context window strategy, and its own response format. The orchestrator coordinates their outputs into a unified response.

### Planned Scope
- Intent classification layer in rag-chat or new rag-orchestrate Lambda
- Agent registry in DynamoDB (agent_id, system_prompt, capabilities)
- Multi-agent conversation memory (track which agent produced each turn)
- Agent coordination protocol for multi-agent tasks
- V3R framework agents (MODERATOR, CODER, DEBUG, etc.) implemented as callable Lambda agents

### Architectural Prerequisites
- Phase 4 GNN must be complete (graph-based knowledge routing is needed for agent specialization)
- Conversation memory (Phase 3) must be stable — multi-agent history is more complex than single-agent

## PHASE 7 — V3R SELF-IMPROVEMENT ENGINE

### Status: PENDING
### Redefined: March 12, 2026

### Original Definition
Science and Engineering Domain Intelligence — ingesting technical domain knowledge.

### Redefined Objective
V3R evaluates its own architecture, identifies performance weaknesses, proposes improvements, implements them, verifies the improvements worked, and repeats. The system engineers itself. This is the cognitive prerequisite for Phase 8 — a system that cannot improve itself cannot reliably build products for others.

### Planned Scope

**Self-Evaluation Pipeline**
- V3R reads its own Architecture Report, Agents Report, Phase Report, and Session Reports from the knowledge base
- V3R computes performance metrics from CloudWatch logs (error rates, latency, memory usage)
- V3R compares actual performance against defined targets
- V3R generates a ranked list of improvement proposals

**Self-Implementation Pipeline**
- For code improvements: V3R writes updated handler code, triggers a rebuild, deploys, runs verification tests
- For prompt improvements: V3R updates agent system prompts in the agent registry
- For architectural improvements: V3R generates a proposal requiring developer approval before execution

**Self-Verification Pipeline**
- After each improvement: V3R runs the standard test suite
- V3R compares pre-improvement and post-improvement metrics
- V3R updates the Phase Report and Agents Report with the results
- Failed improvements are rolled back automatically

### Why Phase 7 Was Redefined
The original Phase 7 (domain intelligence) would have added knowledge — more information for V3R to know. The redefined Phase 7 (self-improvement) adds capability — the ability for V3R to make itself better at knowing and doing. Capability compounds. Knowledge accumulates. For Phase 8 to be reliable, V3R needs capability growth, not just knowledge growth. The self-improvement engine is the mechanism of that capability growth.

### Architectural Prerequisites
- All prior phases must be complete
- V3R reports (Architecture, Agents, Phase, Session) must be ingested into rag-chunks as queryable knowledge
- rag-memory must be stable (Phase 3)
- Agent registry must exist (Phase 5)
- CI/CD-like pipeline must be built: code change → Docker build → ECR push → Lambda deploy → test → verify

## PHASE 8 — AUTOMATED PRODUCT GENERATION

### Status: PENDING

### Objective
The convergence of all prior phases. V3R accepts a raw idea as input and outputs a fully engineered product. Not a draft. Not a suggestion. A product — specified, architected, coded, tested, and deployable.

### What This Requires (in order)
- P1: Vector memory (done)
- P2: Connected pipeline (done)
- P3: Conversational context (active)
- P4: Relational knowledge traversal
- P5: Specialized agent coordination
- P6: Multi-modal input understanding
- P7: Self-improvement for reliability
- P8: Convergence into product generation

### Why Every Prior Phase Is Non-Negotiable
P8 cannot be faked. A system that cannot remember context (P3) cannot maintain product requirements across a multi-turn design session. A system without relational knowledge (P4) cannot identify component dependencies. A system without multi-agent coordination (P5) cannot parallelize engineering tasks. A system without multi-modal understanding (P6) cannot interpret sketches or specifications. A system that cannot improve itself (P7) will degrade in reliability as product complexity grows.

Every phase is a prerequisite. None can be skipped. The decision to build V3R from scratch on free-tier infrastructure is precisely what ensures each phase is built correctly — there is no abstraction layer hiding the engineering from the engineer.

## APPENDIX — PHASE DECISION LOG

| Decision | Phase | Rationale |
|---|---|---|
| Start with text-only ingestion | P1 | Simplest valid modality — validate the pipeline before adding complexity |
| Use Groq for LLM instead of local model | P1 | No GPU available — local LLM inference on CPU is too slow for production |
| Create rag-memory in P2 even though P3 uses it | P2 | Schema creation is cheap — having it ready prevents a table creation step from blocking P3 |
| Separate knowledge and conversation into two tables | P2 | Different access patterns require different optimization strategies |
| Redefine P7 as self-improvement | P3 | Self-improvement is the architectural prerequisite for P8 reliability — domain knowledge alone does not get there |
| All phases sequential, no parallelization | P1 | Each phase is load-bearing for the next — parallel development creates integration debt |
