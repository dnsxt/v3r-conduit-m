SECTION 1: PROJECT OVERVIEW
================================================================================

Project NEXUS — Neurally Extensible Cross-provider Unified System — is the
multi-provider LLM routing and failover layer for the V3R platform.

NEXUS solves V3R's single point of failure: all inference currently routes
through one provider (Groq). If Groq is down, rate-limits, or deprecates
llama-3.1-8b-instant, the entire V3R platform fails. NEXUS eliminates this
by building a dynamic routing layer that:

  1. Maintains a registry of multiple LLM providers in DynamoDB
  2. Routes requests to the optimal provider based on query type
  3. Fails over automatically to backup providers if primary is unavailable
  4. Centralizes all API keys in a single DynamoDB table (v3r-keys)
  5. Allows provider addition/removal without code changes or container rebuilds

NEXUS also addresses the one known billing exposure in V3R: the ECR private
repository costs approximately $0.27/month above free tier. ST-01 migrates
the container image to ECR Public which is permanently free.

NEXUS is the infrastructure intelligence layer. When complete, V3R becomes
resilient, adaptive, and provider-agnostic.

SECTION 8: NEXUS ROUTING LOGIC
================================================================================

Query type classification (to be implemented in lambda_router.py):
  default   — general queries, RAG queries, chat
  code      — queries involving code generation or review
  reasoning — complex multi-step reasoning, analysis

Failover strategy:
  1. Attempt primary provider
  2. If primary fails (timeout, rate limit, error) — try fallback_1
  3. If fallback_1 fails — try fallback_2
  4. If fallback_2 fails — try fallback_3
  5. If all fail — return error with last failure details

Provider failure detection:
  HTTP 429 (rate limit) — immediate failover
  HTTP 500/503 — immediate failover
  Timeout (>30 seconds) — failover
  Empty choices array — failover
