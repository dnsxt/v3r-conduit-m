SECTION A — HOW TO USE THIS DOCUMENT
================================================================================

This document serves two purposes:

1. SPECIFICATION — defines every required section, its content requirements,
   and the rationale behind it. Every field draws from V3R operational history,
   standards analysis, or both. This section (A) is the permanent record of
   why each section exists.

2. TEMPLATE — the blank report structure begins at SECTION B. Copy from
   SECTION B onward when generating a new handoff report. Fill every field.
   No section may be omitted. N/A with a one-line reason is acceptable.
   Never delete a section header — it signals to the receiving AI that the
   section was considered, not forgotten.

THE FUNDAMENTAL PURPOSE OF A HANDOFF REPORT:
  The handoff report is the primary instrument by which V3R solves its own
  stated problem: AI amnesia. A session that begins without the handoff
  report starts from zero. A session that begins with a complete, accurate
  handoff report starts with full operational context. Every omission is
  a reintroduction of the amnesia V3R was built to eliminate.
  This is not documentation for its own sake. It is the memory layer.

THE DIFFERENCE BETWEEN A HANDOFF REPORT AND A PROJECT REPORT:
  Project reports are archival — they record what happened. Handoff reports
  are operational — they enable what happens next. A handoff report must be
  immediately actionable. A receiving AI must be able to execute the first
  instruction of the next session directly from this document. If the
  receiving AI must ask clarifying questions before starting, the handoff
  report failed.

STANDARDS INTEGRATION RATIONALE:

  IEEE 1028-2008 contributes:
    Technical review entry and exit criteria — the handoff report is the
    exit artifact of the outgoing session and the entry artifact of the
    incoming session. Entry criteria (what must be true before the next
    session begins) must be explicitly documented.
    Walkthrough procedure documentation — the step-by-step startup procedure
    is a walkthrough. It must be precise enough to execute without deviation.
    Audit trail — every open item, every active decision, every deferred
    task must be traceable. IEEE 1028 requires that review outputs be
    actionable and assigned.

  NIST SP 800-160 Vol.1 Rev.1 contributes:
    Knowledge Management Process (Section 3.3.6) — knowledge must be
    identified, captured, maintained, and made available. The handoff report
    is the primary knowledge management artifact between sessions.
    Life cycle transition — NIST 800-160 defines transition as a process
    requiring verification that the receiving entity has the information,
    resources, and capabilities to proceed. The handoff report must
    demonstrate this.
    Stakeholder protection — the operator's investment (time, compute cost,
    system integrity) must be protected across session boundaries. An
    incomplete handoff risks both.

  ISO/IEC/IEEE 15288:2023 contributes:
    Transition process requirements — transition must be planned, the
    receiving entity must be enabled, and evidence of successful transition
    must be captured. The receiving AI's ability to execute Step 1 of the
    next session startup constitutes transition evidence.
    Information management — all information needed to continue work must
    be current, accessible, and organized. Drive paths, file locations,
    and system state must be accurate at time of report generation.

  V3R operational history contributes:
    Critical instruction block — every handoff in V3R history begins with
    an explicit instruction to the receiving AI: read all, no summaries,
    no orientation questions, pick up here. This is proven effective and
    must be standardized.
    Partition system — Partition 0 (Framework) and Partition 1 (Briefing)
    must be explicitly seated in every handoff. This is the operational
    memory anchor.
    Active task queue — the most consistent failure mode across all prior
    handoffs was ambiguity about the first action. The active task queue
    with a mandatory "START HERE" marker eliminates this.
    Pre-flight verification — the receiving AI must be able to verify
    system state before making changes. All prior handoffs that included
    verification commands produced faster session starts.
    Open item forward carry — items that could not be resolved in the
    outgoing session must be explicitly marked and carried forward with
    their full context preserved.

HIGHEST-QUALITY REFERENCE REPORT:
  V3R CONDUIT Handoff Report v2.0 (CONDUIT-HR-02, March 15, 2026)
  Rationale: Only report in the library with all of the following: complete
  operational state documentation across all systems, explicit session start
  and session end procedures, full Drive structure map, full local directory
  structure, complete FP registry, lessons learned section, file save
  instructions for the report itself, and a next session startup procedure
  that is executable without clarification. V3R Handoff Report 8 is the
  richest in technical detail (partitioned codebase, active bugs with exact
  fix code, step-by-step task queue) but lacks the system-level operational
  completeness of CONDUIT-HR-02. The standard combines both.
