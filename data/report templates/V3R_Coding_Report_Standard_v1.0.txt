SECTION A — HOW TO USE THIS DOCUMENT
================================================================================

This document serves two purposes:

1. SPECIFICATION — defines every required section, its content requirements,
   and why it exists. This section (A) explains the rationale for each field
   drawn from standards research and V3R operational history.

2. TEMPLATE — the blank report structure begins at SECTION B. Copy from
   SECTION B onward when generating a new coding report. Fill every field.
   No section may be omitted. If a section is not applicable to a session,
   write N/A with a one-line reason. Never delete a section header.

STANDARDS INTEGRATION RATIONALE:

  IEEE 828-2012 contributes:
    Configuration item (CI) identification — every changed file is a CI and
    must be uniquely identified by name, path, version, and baseline state.
    Change traceability — every change traces from trigger → implementation
    → test → verification. No orphaned changes.
    Build and release engineering — the push log is a release record.
    Baseline documentation — the handler reference log is the post-session
    baseline. Future sessions must begin from this baseline.

  NIST SP 800-128 contributes:
    Security impact assessment — every code change must be evaluated for
    whether it touches credentials, IAM, network paths, or data exposure.
    Credential audit — if any session touches API keys, tokens, env vars,
    or DynamoDB credential records, this must be explicitly documented.
    Monitoring evidence — CloudWatch or other evidence confirming the change
    took effect must be captured. "It worked" is not evidence.
    Configuration deviation tracking — if the post-session state differs
    from the expected baseline in any way, it must be flagged.

  V3R operational history contributes:
    Agent gate performance review — drawn from Coding Report 9, which
    identified that the knowledge retrieval and moderator agents failed in
    diagnosable ways that were never captured. Gate performance must be
    reviewed so failures are corrected not repeated.
    Push economy audit — the project has lost hundreds of hours to
    avoidable pushes. Every push must be retrospectively audited.
    Reconstruction test — from V3R's own architectural thesis: a new session
    starting from zero must be able to reconstruct current state from the
    report alone. This requirement is operationally proven.
    PowerShell pattern log — V3R operates on PowerShell 5.1 with constraints
    that cause silent failures. Capturing confirmed working patterns per
    session builds a verified library.

HIGHEST-QUALITY REFERENCE REPORT:
  V3R Coding Report 9 — Claude (March 13, 2026)
  Rationale: Only report with all 10 of the following: table of contents,
  per-attempt failure documentation, root cause analysis on every failure,
  agent gate performance review with named failures, handler reference log,
  push log with batching critique, reconstruction-capable file inventory,
  lessons as actionable entries, API key diagnosis section, and moderator
  self-correction incident. All other reports contain subsets of these.
  Future reports must meet or exceed this standard.
