SECTION A — HOW TO USE THIS DOCUMENT
================================================================================

This document serves two purposes:

1. SPECIFICATION — defines every required section, its content requirements,
   and the rationale for each field. Section A is permanent. It explains the
   "why" behind the standard so that future agents can maintain it intelligently,
   not just mechanically.

2. TEMPLATE — the blank report structure begins at SECTION B. Copy from
   SECTION B onward when generating a new project report. Every section must
   be filled. N/A with a one-line reason is acceptable. Never delete a section
   header — its presence signals the section was considered, not omitted.

THE FUNDAMENTAL PURPOSE OF A PROJECT REPORT:
  The project report is the longitudinal record of V3R. Where the handoff
  report enables the next session and the coding report documents technical
  changes, the project report documents the evolution of the entire system
  across its full arc. A stakeholder, a future AI session, or the developer
  reading report 14 must be able to understand why the system is what it is —
  not just what it is. This requires architectural rationale, roadmap status,
  decision history, and vision — not just current state.

  The project report is also the primary instrument for cross-project
  awareness. V3R, CONDUIT, NEXUS, SENTINEL, and OLYMPIAN are interdependent.
  Every project report must document its relationships to all other active
  projects.

THE DIFFERENCE FROM OTHER REPORT TYPES:
  Coding Report:    Records what changed technically. Narrow focus. Per session.
  Handoff Report:   Enables the next session. Operational focus. Per session.
  Project Report:   Records why and where the project stands. Strategic focus.
                    Per session but with full historical context preserved.

STANDARDS INTEGRATION RATIONALE:

  ISO/IEC/IEEE 15289:2019 contributes:
    Report classification — 15289 defines "report" as a document type that
    "provides a record of something observed, investigated, analyzed, or
    decided." The project report is specifically a Status and Progress Report
    (per Clause 10), which must include: project objectives, scope, status
    against plan, issues and risks, decisions made, and next steps.
    Information item structure — each section of the project report
    corresponds to a 15289 information item. Nothing is documented without
    a standards-backed reason for its existence.

  ISO/IEC/IEEE 15288:2023 contributes:
    Project Management Process (Section 6.3.1) — requires documentation of
    project plan status, work progress measurement, issue tracking, and
    corrective action records.
    Decision Management Process (Section 6.3.8) — decisions must be
    documented with alternatives considered, criteria used, and outcomes.
    Risk Management Process (Section 6.3.4) — risks must be identified,
    analyzed, evaluated, treated, and monitored with full records.
    Technical Review Process — review outputs must state whether the
    subject met requirements, identify defects and deviations, and
    record action items.

  ISO/IEC/IEEE 12207:2017 contributes:
    Software-specific status report content — current version baselines,
    anomalies (bugs), changes implemented, metrics, open items.
    Release documentation — every report produced at a milestone serves
    as a release record for that milestone.
    Configuration audit — the project report documents whether the current
    configuration matches the approved baseline.

  NIST SP 800-160 Vol.1 Rev.1 contributes:
    Stakeholder needs continuity — the vision section ensures the project's
    fundamental purpose is restated in every report, preventing mission drift
    across sessions.
    Mission and business process documentation — why V3R exists must be
    documented as context for every technical decision.
    Life cycle concept documentation — the roadmap section implements this.

  V3R operational history contributes:
    Project overview and vision in every report — V3R Project Report 10
    established the precedent. Even after 14 sessions, the vision was
    restated in full. This prevents mission drift and keeps every AI
    session anchored to the same goal.
    Hardware inventory — V3R's constraint-driven architecture makes the
    hardware inventory a critical decision context. Code written without
    knowing the machine cannot be optimal.
    Forbidden Patterns Registry carry-forward — every report must carry
    the full FP registry so the project report is self-contained.
    Cross-project status — NEXUS reports 4-6 introduced the pattern of
    documenting all active sub-projects in a single report. This is now
    mandatory.
    The "why V3R exists" paragraph — the V3R Project Snapshot Report
    introduced this as an explicit standalone section. It anchors every
    session and every project report.

HIGHEST-QUALITY REFERENCE REPORT:
  V3R Project Report 14 — Claude (March 18, 2026)
  Rationale: The most complete project report in the library by every
  measure. Sections present and at high quality: project overview and
  vision (restated), session summary with explicit issue resolution, NEXUS
  routing verified state with per-provider status, full AWS infrastructure
  with per-function and per-table detail, complete file inventory with line
  counts and NEXUS migration status, full FP registry with 19 entries and
  rationale, operational notes capturing three critical lessons (Groq silent
  failover, Lambda-ECR Public incompatibility, DynamoDB delete-item UUID
  requirement), credentials reference with clear redaction policy, explicit
  next steps, and a future roadmap that traces from completed phases through
  all active and pending projects.

  Closest competitor: V3R Project Report 10 — equally thorough on AWS
  infrastructure and codebase, but predates NEXUS and lacks cross-project
  status, routing state, and the operational lessons density of Report 14.

  V3R Project Snapshot Report — a different format entirely. Goes wide
  rather than deep. Contributes the "document purpose" opening and the
  "living document" classification that are adopted into this standard.
