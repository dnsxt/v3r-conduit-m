Changes from V3.1 (NEXUS-migrated version):

         Q_DOCUMENTATION, Q_INFRASTRUCTURE constants
- ADDED: sqs_send() — publishes JSON payload to FIFO queue
         with MessageGroupId='v3r-pipeline'
- ADDED: sqs_poll() — long-polls queue (20 sec), deletes
         message on receipt, returns parsed body or None
- ADDED: run_pipeline_sqs() — SQS-coordinated pipeline
         Publishes KR, CODER, DEBUG tasks to agent queues
         Polls Q_MODERATOR for results
         Falls back to direct execution on poll timeout
- MODIFIED: lambda_handler() — dual-mode operation:
  Mode 1: SQS trigger (event has 'Records' key)
          Processes agent task from queue record
          Determines agent type from body['agent'] field
          Executes appropriate agent function
          Publishes result back to Q_MODERATOR
  Mode 2: Direct HTTP invocation (standard pipeline)
          Checks use_sqs flag in payload
          Routes to run_pipeline_sqs() or direct execution
          Default: use_sqs=True
- PRESERVED: All agent functions unchanged (moderator, kr,
  coder, debug_gate). All route_llm() calls unchanged.
  All max_tokens and temperature values unchanged.
  get_rag_context() and get_graph_context() unchanged.
  Return shape unchanged.

Architecture note: In current implementation, orchestrator
acts as both MODERATOR (publisher) and agent executor
(consumer). SQS provides coordination infrastructure for
future dedicated per-agent Lambda functions. Fallback to
direct execution ensures pipeline never fails due to SQS
timeout.

3. BUGS ENCOUNTERED AND RESOLVED
═══════════════════════════════════════════════════════════

BUG-003 — EventBridge Scheduler role error
Symptom: ValidationException — execution role must allow
         EventBridge Scheduler to assume the role
Root cause: Lambda execution role passed as scheduler role —
            trust relationship only covers lambda.amazonaws.com
Fix: Created dedicated v3r-scheduler-role with
     scheduler.amazonaws.com trust
Status: RESOLVED

BUG-004 — Docker build 403 Forbidden on ECR Public base image
Symptom: failed to resolve source metadata for
         public.ecr.aws/lambda/python:3.12 — 403 Forbidden
Root cause: ECR Public auth token expired — Docker not
            authenticated to pull from public.ecr.aws
Fix: aws ecr-public get-login-password --region us-east-1 |
     docker login --username AWS --password-stdin public.ecr.aws
     Run before any buildx command after 12+ hour gap
Status: RESOLVED — FP-021 established
