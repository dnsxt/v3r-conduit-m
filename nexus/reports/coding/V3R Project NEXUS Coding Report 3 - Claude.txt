PURPOSE:
  NEXUS provider routing utility. Single shared module imported by all V3R
  Lambda handlers. Exposes one public function: route_llm(). Resolves route_id
  to provider order via v3r-routing DynamoDB table. Selects best available
  key from v3r-keys table using failover strategy. Executes call via OpenAI-
  compatible HTTP adapter. Returns normalized text response. Caller never
  knows which provider executed the request.

IMPORTS:
  json, boto3, http.client, datetime.date
  All standard library except boto3 — confirmed present in container.
  No new wheels required.

KEY FUNCTIONS:

  get_route(route_id)
    Fetches routing record from v3r-routing table by route_id.
    Falls back to 'default' route if route_id not found.

  get_key(provider)
    Fetches provider key record from v3r-keys table by key_id.
    Returns full record including api_key, base_url, model, status,
    daily_limit, requests_today, last_reset.

  reset_if_new_day(provider, key_record, today)
    Lightweight daily reset. Checks last_reset against today's date.
    If different, resets requests_today to 0 and updates last_reset.
    EventBridge midnight reset is primary mechanism — this is backup.

  increment_usage(provider)
    Increments requests_today counter after successful call.
    Uses DynamoDB UpdateExpression atomic increment.

  call_openai_compat(key_record, system_prompt, user_prompt, max_tokens,
                     temperature)
    OpenAI-compatible HTTP adapter.
    Uses http.client.HTTPSConnection — FP-002 compliant.
    base_url stored without https:// prefix — HTTPSConnection requires
    host only. Constructs standard OpenAI chat completions payload.
    Returns content string from choices[0].message.content.

  route_llm(route_id, system_prompt, user_prompt, max_tokens=1024,
            temperature=0.3)
    Main entry point for all V3R handlers.
    1. Resolves route_id to provider order via v3r-routing.
    2. Iterates provider order. Skips exhausted, paused, or missing keys.
    3. Resets daily counter if new day.
    4. Checks requests_today against daily_limit. Marks exhausted if over.
    5. Executes call_openai_compat on first available provider.
    6. Increments usage counter on success.
    7. Returns text response or routing error string if all providers fail.

DESIGN NOTES:
  - Gemini adapter reserved for future extension — not yet implemented.
    Gemini uses different request/response schema than OpenAI-compatible
    providers. When Gemini is registered, a call_gemini() adapter function
    will be added and the adapter field in v3r-keys will route to it.
  - daily_limit field defaults to 9999 if not present in key record.
    This prevents KeyError on keys that predate the daily_limit field.
  - Exception handling on call_openai_compat continues to next provider
    rather than raising. This makes the failover chain resilient to
    transient network errors and provider-side failures.
