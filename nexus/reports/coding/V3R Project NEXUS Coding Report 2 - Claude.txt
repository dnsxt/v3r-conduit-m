NEW FORBIDDEN PATTERNS ADDED THIS SESSION
================================================================================

FP-020: Hardcoding provider API keys in Lambda handler code
  All provider API keys must be retrieved from DynamoDB v3r-keys table
  at Lambda runtime. Never store keys in handler code, environment
  variables, Dockerfile, or any config file pushed to Drive.
  Correct pattern:
    dynamodb.get_item(
      TableName='v3r-keys',
      Key={'key_id': {'S': 'provider_name'}}
    )['Item']['api_key']['S']
  Why: Keys rotate. Central storage means one update propagates everywhere.
  Hardcoded keys require container rebuild to update — 67 minute push cycle.

FP-021: Assuming provider API endpoint format matches Groq
  Each provider has its own API endpoint and auth pattern:
    Groq:      api.groq.com — http.client.HTTPSConnection
               POST /openai/v1/chat/completions
    Cerebras:  api.cerebras.ai — OpenAI-compatible
               POST /v1/chat/completions
    Mistral:   api.mistral.ai — own REST API
               POST /v1/chat/completions
    Fireworks: api.fireworks.ai — OpenAI-compatible
               POST /inference/v1/chat/completions
  Always verify endpoint path and auth header format before coding each
  provider integration. Do not assume /openai/v1/chat/completions works
  for all providers.
