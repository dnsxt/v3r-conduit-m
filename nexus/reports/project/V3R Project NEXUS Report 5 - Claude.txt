SECTION 1: PROJECT OVERVIEW AND VISION
================================================================================

V3R is a Cloud-Native GNN-integrated Multi-Agent Multi-LLM Industrial Science
and Intelligence Engineering Platform. It is not a chatbot. It is not a tool.
It is the foundational architecture for an AI system that accepts raw ideas as
input and outputs fully engineered products.

The core problem V3R solves is AI amnesia — the complete absence of persistent
memory in every current AI system. V3R's persistent memory layer eliminates
this permanently. Knowledge ingested today is retrieved tomorrow. Conversations
accumulate into a growing intelligence base. Engineering decisions are
remembered across sessions.

As of this report, V3R is self-aware, operationally complete through Phase 8,
synchronized to Google Drive via CONDUIT, and actively mid-deployment of
Project NEXUS — a multi-provider LLM routing layer that eliminates single-
provider dependency across all V3R Lambda functions.

V3R is being built entirely on free-tier AWS infrastructure by a single
developer on constrained personal hardware with no admin privileges.

SECTION 4: PROJECT NEXUS — ACTIVE
================================================================================

Project NEXUS — Neurally Extensible Cross-provider Unified System — is the
multi-provider LLM routing layer for V3R. NEXUS eliminates single-provider
dependency by routing all LLM calls through a failover chain: groq ->
cerebras -> mistral/fireworks depending on route tier.

NEXUS ARCHITECTURE:
  - lambda_router.py: Shared routing utility imported by all Lambda handlers.
    Exposes single function: route_llm(route_id, system, user, max_tokens,
    temperature). Resolves route via v3r-routing table. Selects best available
    key via v3r-keys table. Returns normalized text — handlers never know
    which provider executed.
  - v3r-routing DynamoDB table: 8 routes, each with primary + 3 fallbacks.
  - v3r-keys DynamoDB table: 4 providers with api_key, base_url, model,
    daily_limit, requests_today, last_reset, status.
  - EventBridge rule: Midnight daily reset of requests_today counters
    (implementation status: per roadmap, not yet confirmed complete this
    session).

         v3r-routing and v3r-keys tables populated. All functions tested.
  ST-02: ACTIVE — DynamoDB provisioned capacity for rag-chunks and rag-graph.

SECTION 14: OPERATIONAL NOTES AND LESSONS LEARNED
================================================================================

NEXUS BASE_URL FORMAT:
  lambda_router.py uses http.client.HTTPSConnection which requires host only —
  no https:// prefix. base_url must be stored as plain hostname (api.groq.com)
  not as full URL. All 4 providers corrected post-deployment via DynamoDB
  update-item. This is now the documented standard for v3r-keys records.

FIRST TEST ROUTING ERROR:
  The "all providers exhausted" error on the first rag-chat test was caused by
  missing base_url fields in v3r-keys, not by provider API failures. After
  correction, all providers became available. Lesson: verify v3r-keys records
  completeness before declaring a routing failure a provider issue.

DOCKER DESKTOP DAEMON:
  Two consecutive build failures with "failed to connect to docker API at
  npipe://..." occurred before the third attempt succeeded. Root cause: Docker
  Desktop daemon was starting up. The ECR re-login between the second and
  third attempt was correct procedure regardless. Always confirm Docker Desktop
  is fully running before build — icon in system tray should show no loading
  animation.

PUSH TIME CONFIRMED:
  The March 16 push completed in 24 seconds — handler-only cached layers.
  Only steps 12-18 (the 7 handler COPY commands) were non-cached. This is
  significantly faster than the 67-minute handler-only push documented in
  Report 11. The difference may be context-transfer size (4.75 MB this push).
  67 minutes remains the documented worst-case baseline.

TIMESTAMP VIOLATION:
  Prior responses this session used a static 03:14:26 timestamp. Corrected
  March 17, 2026. Time tool called per response going forward.

GROQ PATH NOTE:
  lambda_router.py calls /v1/chat/completions hardcoded. Groq's documented
  path is /openai/v1/chat/completions. The verified test succeeded — either
  Groq accepts both paths, or another provider (cerebras/mistral) handled the
  call. Flag for verification in a future session.
