SECTION 1: PROJECT OVERVIEW AND VISION
================================================================================

V3R is a Cloud-Native GNN-integrated Multi-Agent Multi-LLM Industrial Science
and Intelligence Engineering Platform. It is not a chatbot. It is not a tool.
It is the foundational architecture for an AI system that accepts raw ideas as
input and outputs fully engineered products.

The core problem V3R solves is AI amnesia — the complete absence of persistent
memory in every current AI system. V3R's persistent memory layer eliminates
this permanently. Knowledge ingested today is retrieved tomorrow. Conversations
accumulate into a growing intelligence base. Engineering decisions are
remembered across sessions. The system becomes more capable with every
interaction.

As of this report, V3R is fully operational across all 7 Lambda handlers,
all 4 LLM providers are routing correctly including Groq (previously silently
failing due to wrong endpoint path), CONDUIT is synchronized, and all known
infrastructure issues from the exhaustive AWS condition report have been
resolved.

V3R is being built entirely on AWS infrastructure by a single developer on
constrained personal hardware with no admin privileges.

SECTION 12: OPERATIONAL NOTES AND LESSONS LEARNED
================================================================================

GROQ SILENT FAILOVER:
  The most dangerous failure mode in NEXUS is a silent provider failure.
  lambda_router catches all exceptions and moves to the next provider with
  no logging. This means a broken primary can fail on every single call
  for days without any visible error — only the requests_today counter
  being 0 reveals it. Monitoring strategy: check v3r-keys requests_today
  values periodically. A provider showing 0 when it is primary on active
  routes is a red flag requiring immediate investigation.

LAMBDA CANNOT PULL FROM ECR PUBLIC:
  Lambda's update-function-code API accepts only private ECR URIs.
  ECR Public is for public distribution of container images — not for
  Lambda deployment. The AWS console and PowerShell both correctly rejected
  public URIs. ECR Public has no role in the V3R Lambda architecture.
  Future NEXUS sessions should not attempt ECR Public migration for Lambda.

IAM PATH ACTIONS:
  ECR Public IAM actions are in the ecr-public: namespace.
  ecr-public:BatchGetImage and ecr-public:GetDownloaderUriForLayer do not
  exist. Correct actions: ecr-public:GetAuthorizationToken,
  ecr-public:GetDownloadUrlForLayer, sts:GetServiceBearerToken.
  Moot for Lambda — documented for reference only.

DYNAMODB DELETE-ITEM REQUIRES ACTUAL PK:
  delete-item requires the true partition key value (UUID in rag-chunks),
  not the value of a non-key field like source. Always scan with a
  FilterExpression to retrieve the chunk_id UUID before attempting deletion.
