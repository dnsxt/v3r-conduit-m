1. EXECUTIVE SUMMARY


Project NEXUS is the next phase of V3R — a distributed multi-provider intelligence architecture that transforms V3R from a single Groq-backed RAG system into a self-routing mesh of specialized AI providers, each assigned to the task tier it executes best, coordinated by a permanent zero-cost AWS infrastructure that scales horizontally through account multiplication rather than paid upgrades.
Three rounds of independent research, cross-validated and run through V3R's own knowledge base, identified five confirmed zero-cost AI providers, four critical AWS always-free services previously underutilized by V3R, one confirmed billing exposure requiring immediate remediation, and a provider plug-in architecture that allows new AI models to be registered and activated without any code changes to deployed Lambda handlers.
The result is a system that becomes more capable with every new provider registered, more resilient with every additional account created, and more intelligent with every interaction — all at permanent zero cost.
2. PROJECT ORIGIN — WHAT V3R REVEALED


V3R was built to solve AI amnesia — the complete absence of persistent memory in every standard AI session. By Phase 8, V3R had solved that problem. It had persistent RAG memory, a GNN knowledge graph, multi-agent orchestration, self-knowledge domain intelligence, and an automated product specification generator — all running on AWS free tier, zero cost, zero compromise.
What Phase 8 revealed was a different problem. V3R's entire intelligence pipeline runs on a single LLM provider — Groq, with a single model — llama-3.1-8b-instant. Every agent from MODERATOR to SYSTEM DEBUG executes on the same 8B parameter model. The simulation labels in the Framework constrain agent behavior but cannot raise the capability ceiling of the underlying model. A system that simulates a reasoning specialist on a general-purpose fast inference model will always produce outputs bounded by that model's ceiling.
V3R's architectural thesis — that the system becomes more capable with every interaction — cannot be fully realized if every agent is executing on the same hardware. The next evolution is not more memory. It is better thinking. Project NEXUS is the answer.
3. THE PROBLEM STATEMENT


Single Provider Dependency: All V3R intelligence routes through Groq. If Groq changes rate limits, deprecates the model, or experiences downtime, V3R is completely offline. No fallback exists.
Uniform Capability Ceiling: All six V3R agents execute on the same 8B model. The SYSTEM DEBUG agent — designed for deep chain-of-thought validation — operates at the same capability level as the simple retrieval formatter. This is architectural waste.
No Provider Abstraction: Every new LLM provider requires code changes across multiple Lambda handlers. HTTP connection patterns, authentication headers, and response schemas differ per provider. Integration cost per new provider is high.
Single Account Rate Pool: All V3R requests draw from one Groq account's daily limits. 14,400 RPD sounds large until a complex multi-pass pipeline consumes 30+ calls per user request. Five concurrent users exhaust the daily limit in hours.
ECR Billing Exposure: V3R's container image (3,184 MB) sits in a private ECR repository. After 12 months, the 500MB free storage tier expires and billing begins. An always-free public ECR alternative exists but has not been implemented.
No Load Distribution: V3R has no mechanism to distribute requests across multiple providers or accounts. All load concentrates on a single point. Homeostasis — the self-regulating state where no component is overloaded — is not achievable with current architecture.

4. THE NEXUS THESIS


A speaker crossover routes frequency bands to the driver architecturally suited for them. Tweeters handle highs. Woofers handle lows. Not because any single driver is insufficient — but because specialization produces superior aggregate output with less strain per component. The crossover circuit is the intelligence of the system. It does not generate sound. It routes it correctly.
NEXUS is V3R's crossover circuit. Different task types route to the AI best suited for them. Load distributes across multiple free-tier accounts so no single account is exhausted. Every provider operates within its capability sweet spot. The aggregate intelligence of the system exceeds what any single provider could deliver — at zero cost, permanently.
The NEXUS thesis has three components:
Tier Routing. Tasks are classified by complexity tier before execution. Each tier maps to the provider best suited for that tier at zero cost. Simple, high-volume tasks route to the fastest free provider. Complex reasoning routes to the strongest free reasoning model. Validation routes to the most rigorous free model available.
Provider Abstraction. All handlers call a single internal function: llm_call(tier, system, prompt). The routing layer resolves tier to provider, formats the request per that provider's API, and returns a normalized response. Adding a new AI requires no handler code changes — only a new registry record.
Homeostasis. A DynamoDB routing table tracks real-time usage per key. When a key approaches its daily limit, the router automatically selects the next available key in the same tier. The system self-regulates. No manual intervention. No downtime. No billing surprises.
5. RESEARCH METHODOLOGY — THREE ROUND ANALYSIS


NEXUS was not designed from assumption. It was derived from three independent rounds of research, each sourced from different references, each output run through V3R's own knowledge base for compatibility and constraint checking, each round then compared against the prior rounds to identify conflicts, correct errors, and refine findings.
Round 1: Primary provider documentation. Groq rate limit docs, AWS free tier official pages, HuggingFace tier documentation. Established baseline numbers and initial architecture concepts. Identified IoT Core as agent coordination layer (later corrected). Missed Cerebras, OpenRouter, and Mistral entirely.
Round 2: Independent community references and third-party AWS billing analysis. Discovered July 2025 AWS free tier restructure. Corrected IoT Core classification (12-month only, not always-free). Identified ECR Public as billing exposure solution. Replaced IoT Core recommendation with SQS (always-free). Identified AWS Organization aggregation gotcha that invalidates multi-account strategy if accounts are joined to an Organization.
Round 3: Alternative provider sources and community benchmarks. Discovered Cerebras — confirmed fastest free inference provider at 1,800 tokens/second on Llama 3.1 8B, no credit card, no waitlist. Discovered OpenRouter — single unified endpoint for 290+ models, OpenAI-compatible, free tier available. Confirmed Mistral Experiment plan (phone number only, 1B tokens/month). Introduced the provider abstraction layer concept as the architectural solution to multi-provider integration complexity.
All three rounds were looped through V3R's RAG knowledge base and knowledge graph for constraint checking against the platform's known architecture, forbidden patterns, and hardware limitations before findings were accepted into this report.

8. THE PROVIDER PLUG-IN SYSTEM


Adding a new AI provider to V3R currently requires code changes across every handler that makes LLM calls — new HTTP connection patterns, new authentication headers, new response parsing. NEXUS eliminates this entirely through a provider registry and adapter pattern.
The v3r-keys DynamoDB table is the provider registry. Schema:
Field
Type
Description
key_id (S)
Partition Key
Unique identifier e.g. groq_01, cerebras_02, gemini_debug_01
provider (S)
Attribute
groq | gemini | cerebras | mistral | openrouter
model (S)
Attribute
Exact model string — e.g. llama-3.1-8b-instant
api_key (S)
Attribute
The API key value — stored here, never hardcoded in handlers
base_url (S)
Attribute
Provider API base URL — e.g. api.groq.com, api.cerebras.ai
tier (S)
Attribute
speed | quality | debug | volume | batch
daily_limit (N)
Attribute
RPD cap for this key
requests_today (N)
Attribute
Rolling counter — incremented by router on each use
last_reset (S)
Attribute
ISO timestamp of last midnight reset
status (S)
Attribute
active | paused | exhausted
adapter (S)
Attribute
openai_compat | gemini — determines HTTP request format


Adding a new AI — four steps:
Register account with new provider. Get API key.
Add one row to v3r-keys table via PowerShell (WriteAllText + aws dynamodb put-item).
If provider is OpenAI-compatible (Groq, Cerebras, Mistral, OpenRouter, xAI are all compatible): no new adapter needed. Set adapter field to openai_compat.
If provider uses a different API schema (Gemini): add one adapter function to the routing utility. One push required.
The commented-out template pattern: the routing utility file contains one commented-out template record at the top, with each field on its own line preceded by #. To activate a new provider, fill in the values, remove the #, run the PowerShell put-item command. No handler files are touched. No Docker push required. The new provider is live immediately.

11. HOMEOSTASIS MECHANISM


Homeostasis is the state where no component is overloaded, degraded requests are automatically rerouted, and aggregate capability remains stable regardless of individual component state. NEXUS achieves homeostasis through the routing utility function and v3r-keys table.
Routing Algorithm:
1. Handler calls route_llm(tier, system, prompt).
2. Router queries v3r-keys: filter by tier=requested_tier AND status=active.
3. Sort results by (requests_today / daily_limit) ascending — least utilized key first.
4. Select first key. Check if last_reset date is today. If not, reset requests_today to 0.
5. Increment requests_today by 1. If requests_today >= daily_limit, set status=exhausted.
6. If no active keys remain in requested tier, fall back to next tier down.
7. Format request per adapter type (openai_compat or gemini). Execute call.
8. Return normalized text response to handler. Handler never knows which provider executed.
The routing utility is a shared Lambda layer — one Python file, ~50 lines, deployed once, imported by all handlers. Updates to the routing logic deploy in a single push. The v3r-keys table is the live state. The routing utility is the decision engine. Together they are the crossover circuit.
EventBridge Scheduler — Midnight Reset:
An EventBridge rule fires daily at midnight UTC. It invokes a lightweight Lambda function that scans v3r-keys and resets requests_today to 0 and status to active for all keys. This is the heartbeat of homeostasis. EventBridge Scheduler is always-free at 14M invocations/month. The reset Lambda executes in under one second.

17. CONSTRAINTS AND NON-NEGOTIABLES


ZERO COST: No provider, service, or infrastructure component may require payment at any level of NEXUS operation. If a free tier has a credit card requirement at signup but never charges within free limits, it is acceptable. If a service charges when limits are exceeded, NEXUS must be architecturally incapable of exceeding those limits.
NO AWS ORGANIZATIONS: Multiple AWS accounts must remain completely independent. Joining any NEXUS account to an AWS Organization aggregates always-free limits across all member accounts, destroying the capacity multiplication strategy entirely.
PAID PLAN ONLY FOR NEW AWS ACCOUNTS: Free Plan accounts auto-close after 6 months. All new AWS accounts created for NEXUS must be Paid Plan. Credit card required at signup but never charged within always-free limits.
GROQ AS BACKBONE: Gemini, Cerebras, Mistral, and OpenRouter are supplementary. V3R must remain fully operational if all supplementary providers become unavailable simultaneously. Groq is the backbone. It is never a secondary fallback — it is the guaranteed floor.
